The CLI handles the file work you shouldn't hand to an LLM: exporting translated locale files into your repo and importing source strings back into Kalem. Use it locally during development and in CI to keep your repository and Kalem in sync.

## Typical workflow

```bash
kalem login --url https://<api-url>/ --key <your-api-key>
kalem init --project proj_123 --format json-nested --path "locales/{locale}.json"
kalem pull   # download translations into repo
kalem push   # upload source strings + any local translations
```

`kalem init` writes a `.kalem.json` config file to your working directory. Commit this file — it records the project ID, file format, and path pattern so you don't have to repeat the flags.

## `push` semantics

`push` reads your locale files and syncs them to Kalem. Understanding what it does and doesn't do is important before you run it in CI.

- **Base-locale file** — upserts keys and sets their base values. New keys are created; existing keys are updated if the value has changed.
- **Other locale files** — set translations for keys that already exist. A locale file cannot create new keys; only the base locale can.
- **Keys absent from the base-locale file** are **deprecated**, not deleted. Kalem retains them for audit history and automatically reactivates them if they reappear in a future push.

Three flags modify this behavior:

| Flag | Effect |
|---|---|
| `--prune` | Hard-deletes keys that are absent from the base-locale file instead of deprecating them. Use with care — this is irreversible. |
| `--dry-run` | Prints exactly what would be created, updated, or deprecated without writing anything. Safe to run at any time. |
| `--check` | Runs QA checks after a successful push and exits non-zero if any error-severity findings are reported. |

## `pull` semantics

`pull` downloads the current state of each locale from Kalem and writes locale files to the paths defined in `.kalem.json`.

By default, `pull` writes only **approved values** — translations that have passed review. In-progress and draft work stays in Kalem and doesn't ship to your repository.

Two flags modify this behavior:

| Flag | Effect |
|---|---|
| `--working` | Includes draft (unapproved) translations in addition to approved values. Useful when previewing in-progress work locally. |
| `--exclude-stale` | Skips translations that have been marked stale because their source string changed. The stale values are omitted from the output file entirely. |

```bash
kalem pull --working               # includes unapproved in-progress translations
kalem pull --exclude-stale         # omits stale translations from the output
kalem pull --working --exclude-stale  # both
```

<Warning>
  `pull` is not a perfect round-trip. Kalem may store richer metadata (comments, plural variants, context) that the locale file format can't represent. Always treat the locale files as a build artifact of Kalem — the source of truth lives in Kalem, not in the files.
</Warning>

## Using the CLI in CI

In CI environments, skip `kalem login` and use environment variables instead of a stored `auth.json` file:

| Variable | Purpose |
|---|---|
| `KALEM_API_URL` | Your Kalem API base URL |
| `KALEM_API_KEY` | API key with appropriate scopes |

A typical CI gate that validates your source strings and checks translation quality before merging:

```bash
kalem push --dry-run --check
```

This pushes nothing but reports what would change and exits non-zero if QA checks find errors — making it safe to run on every pull request.

A GitHub Actions example is available at [`examples/translations-sync.yml`](../examples/translations-sync.yml) in the Kalem repository.

### Exit codes

| Code | Meaning |
|---|---|
| `0` | Success — no errors |
| `1` | QA error — one or more error-severity findings |
| `2` | Usage error — bad flags, missing config, or invalid arguments |
| `3` | API or network error — Kalem was unreachable or returned an unexpected response |

### Machine-readable output

Pass `--json` to any command to receive structured JSON output on stdout. This is useful when you want to parse results in a script or forward them to another tool:

```bash
kalem push --dry-run --check --json
```
