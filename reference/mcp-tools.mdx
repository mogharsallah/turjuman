Kalem's MCP server exposes tools that cover every management operation — from creating projects and managing keys to running QA checks and administering users. Every tool call is authenticated with `Authorization: Bearer <api-key>`. You can mount the server directly in any MCP-compatible agent and start managing your translations programmatically.

## Projects & Locales

| Tool | Purpose |
|---|---|
| `list_projects` | List all projects you can access |
| `get_project` | Fetch a single project by ID |
| `create_project` | Create a new project (requires `baseLocale`) |
| `update_project` | Update project settings (name, base locale, etc.) |
| `list_locales` | List all locales configured for a project |
| `add_locale` | Add a new target locale to a project |

## Keys

| Tool | Purpose |
|---|---|
| `list_keys` | List keys in a project (paged: `limit` + `nextCursor`) |
| `search_keys` | Full-text search across key names and descriptions (paged) |
| `get_key` | Fetch a single key by ID or name |
| `create_key` | Create a key — optionally set `description`, `tags`, `maxLength`, `plural` |
| `update_key` | Update key metadata (description, tags, maxLength, plural) |
| `delete_key` | Hard-delete a key and all its translations — requires `confirm: true` |

## Translations

| Tool | Purpose |
|---|---|
| `get_translations` | Fetch translations by key (all locales) or by locale (paged) |
| `list_untranslated` | List keys that have no value for a given locale (paged) |
| `list_stale` | List translations whose source value has changed since they were written (paged) |
| `set_translation` | Write a single working `value` for a key/locale pair |
| `bulk_set_translations` | Write up to 500 translation entries in one call; unknown keys are silently skipped |
| `set_translation_status` | Advance the workflow status of a translation (e.g. `approved`) |

## Glossary & Translation Memory

| Tool | Purpose |
|---|---|
| `list_glossary` | List all glossary terms for a project |
| `add_glossary_term` | Add a new term to the glossary |
| `update_glossary_term` | Update an existing glossary term |
| `remove_glossary_term` | Remove a term from the glossary |
| `lookup_translation_memory` | Find prior phrasings for a value — searches exact, normalized, and bigram-fuzzy matches |

## Quality (QA)

| Tool | Purpose |
|---|---|
| `run_qa_checks` | Run advisory checks — accepts `{ projectId, locale?, checks?, slot? }` |
| `get_qa_config` | Read the per-project QA configuration |
| `set_qa_config` | Write the per-project QA configuration (enable/disable checks, remap severities, add ignores) |

## Webhooks

| Tool | Purpose |
|---|---|
| `list_webhooks` | List all webhooks registered for a project |
| `add_webhook` | Register a new webhook — the signing secret is returned only once at creation time |
| `remove_webhook` | Remove a registered webhook |

## Administration

| Tool | Purpose |
|---|---|
| `list_users` / `create_user` | List or create users in your organisation |
| `set_user_role` | Set a user's global role — OWNER-gated for privileged roles |
| `list_members` / `add_member` / `set_member_role` / `remove_member` | Manage per-project membership |
| `create_api_key` / `list_api_keys` / `revoke_api_key` | Create, list, and revoke API keys |

## Paging

Tools that return lists support cursor-based pagination. Pass `limit` (default `100`, max `200`) to control page size. When there are more results, the response includes a `nextCursor` field — pass it back as `cursor` in your next call to retrieve the following page. Continue until `nextCursor` is absent or `null`.

```json
// First call
{ "projectId": "proj_xxx", "limit": 50 }

// Subsequent call
{ "projectId": "proj_xxx", "limit": 50, "cursor": "eyJpZCI6Mn0" }
```

## Toolset scoping

You can limit which tools are exposed when you mount the MCP server. This is useful for read-only agents, narrowly scoped workflows, or least-privilege API keys.

### Scoping by API key

The tools available to an agent are automatically constrained by the permissions of the API key used to authenticate:

- A **read-only key** can only invoke read tools (`list_*`, `get_*`, `search_*`, `lookup_*`).
- A key with global **MEMBER** role cannot invoke org-level administration tools.

### Scoping by URL parameters

You can further restrict tools at mount time using query parameters on the MCP server URL:

| Parameter | Value | Effect |
|---|---|---|
| `tools` | Comma-separated tool names | Expose only the listed tools |
| `groups` | Comma-separated group names | Expose only tools in the listed groups |

Available group names: `projects`, `keys`, `translations`, `glossary`, `webhooks`, `qa`, `admin`, `read`.

The `read` group is a convenience alias that includes all list/get/search/lookup tools across every resource type.

```
https://<your-mcp-url>/?groups=read,translations
https://<your-mcp-url>/?tools=list_keys,get_key,set_translation
```

## Tool behavior hints

These hints are declared in each tool's MCP schema so compatible agent frameworks can make safer decisions (e.g. confirming before a destructive action).

| Property | Tools where set | Meaning |
|---|---|---|
| `readOnlyHint: true` | All `list_*`, `get_*`, `search_*`, `lookup_*` | Tool does not modify any server state |
| `destructiveHint: true` | `delete_key`, `remove_webhook`, `revoke_api_key`, `remove_member`, `remove_glossary_term` | Tool permanently removes data |
| `idempotentHint: true` | `set_translation`, `set_translation_status`, `set_qa_config`, `set_user_role`, `set_member_role` | Calling the tool multiple times with the same arguments produces the same result |

---

<Note>
When a tool encounters a validation or API error, it returns the error message as MCP `content` with `isError: true` rather than throwing. This means your agent can read the error description and adjust its next action without the tool call itself failing the chain.
</Note>
