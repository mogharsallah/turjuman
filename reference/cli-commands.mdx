The `kalem` CLI is purpose-built for deterministic locale file work: it pulls translations from Kalem and writes them to your repository as locale files, and it pushes locale files back to Kalem. Everything else — creating projects, managing keys, running QA from an agent, administering users — goes through the MCP server or the REST API.

<Note>
Deploy commands (`deploy`, `status`, `teardown`, `bootstrap`) live in the separate `@kalem/deploy` package (binary: `kalem-deploy`) and are not part of the core CLI documented here.
</Note>

---

## Installation

The CLI source lives in `packages/cli` in the Kalem monorepo. To build and run it locally:

```bash
npm run build
node packages/cli/dist/index.js <command>
```

When published, the binary is available as `kalem`:

```bash
kalem <command>
```

## Configuration

Kalem uses two configuration files with different scopes and commit policies.

### `kalem.config.json` — project configuration

This file lives in your repository root, is created by `kalem init`, and **should be committed to version control**. It tells the CLI which project to work with and how to map Kalem namespaces to local file paths.

```json
{
  "projectId": "proj_xxx",
  "targets": [
    { "format": "json-nested",     "path": "locales/{locale}.json",                          "namespace": "default" },
    { "format": "android",         "path": "android/values-{locale}/strings.xml" },
    { "format": "ios-strings",     "path": "ios/{locale}.lproj/Localizable.strings" },
    { "format": "ios-stringsdict", "path": "ios/{locale}.lproj/Localizable.stringsdict" }
  ]
}
```

Each entry in `targets` maps a file format to a path pattern. The `{locale}` placeholder in `path` is substituted with each locale code when reading or writing files (e.g. `locales/fr.json`). The `namespace` field is optional and defaults to `"default"`.

### `~/.kalem/auth.json` — machine credentials

This file stores your API URL and key for the current machine. It is created by `kalem login` and **must not be committed to version control**.

```json
{
  "apiUrl": "https://kalem.example.com",
  "apiKey": "kalm_live_xxxxxxxxxxxxxx"
}
```

### CI environment variables

In CI pipelines, use environment variables instead of `auth.json`:

| Variable | Purpose |
|---|---|
| `KALEM_API_URL` | Base URL of your Kalem instance |
| `KALEM_API_KEY` | API key for authentication |

Environment variables take precedence over `~/.kalem/auth.json` when both are present.

## Commands

| Command | What it does |
|---|---|
| `login --url <url> --key <key>` | Save credentials to `~/.kalem/auth.json` for this machine |
| `init --project <id> [--format <fmt>] [--path <pattern>] [--namespace <ns>]` | Scaffold a `kalem.config.json` in the current directory |
| `projects` | List all projects your API key can access |
| `locales` | List the locales configured for the current project |
| `formats` | List all supported file format IDs |
| `pull [--working]` | Download translations from Kalem and write locale files to disk |
| `build` | Alias for `pull` — useful in build scripts |
| `push [--prune] [--dry-run] [--check]` | Read locale files and upload translations to Kalem |
| `check [--locale <code>] [--check <id...>] [--approved]` | Run advisory QA checks and print the findings report |

### `pull` options

| Flag | Effect |
|---|---|
| `--working` | Pull the current working (draft) values instead of the last-approved values |

### `push` options

| Flag | Effect |
|---|---|
| `--prune` | Delete keys in Kalem that are absent from the pushed files |
| `--dry-run` | Compute the diff and print what would change without writing anything |
| `--check` | After pushing, run QA checks and exit with code `1` if there are error-severity findings |

### `check` options

| Flag | Effect |
|---|---|
| `--locale <code>` | Restrict checks to a single locale (may be repeated) |
| `--check <id>` | Run only the specified check(s) (may be repeated) |
| `--approved` | Only check translations that have been approved (skips draft values) |

## `--json` flag

Every command accepts `--json` to produce machine-readable output. When `--json` is set:

- Standard output contains a single JSON object or array.
- Progress messages and spinners are suppressed.
- Diagnostic messages (warnings, errors) are written to **stderr** only.

This flag is designed for CI pipelines and scripting where you need to parse command output programmatically.

```bash
kalem pull --json | jq '.locales'
kalem check --json > findings.json
```

## Exit codes

| Code | Meaning |
|---|---|
| `0` | Success — command completed without issues |
| `1` | QA findings with error severity were found |
| `2` | Usage error — bad flags, missing `kalem.config.json`, or not logged in |
| `3` | API or network error — the server returned an error response or was unreachable |

<Note>
API error responses include a `requestId` field. You can provide this ID when reporting issues — it correlates directly with the corresponding entry in your Kalem server logs.
</Note>

## Supported formats

Run `kalem formats` to get the live list from your Kalem instance. The currently supported format IDs are:

| Format ID | File type |
|---|---|
| `json-nested` | `.json` |
| `json-flat` | `.json` |
| `yaml` | `.yaml` / `.yml` |
| `arb` | `.arb` |
| `properties` | `.properties` |
| `csv` | `.csv` |
| `android` | `strings.xml` |
| `ios-strings` | `.strings` |
| `ios-stringsdict` | `.stringsdict` |

See the [File Formats reference](/reference/file-formats) for a full description of each format including plural handling.
