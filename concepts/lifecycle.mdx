Continuous localization means your translations move with your code rather than in a separate, infrequent release cycle. Just as you run tests on every commit, you push new source strings to Kalem on every relevant change, let the translation pipeline run, and pull approved translations back in before shipping. Kalem is designed around this loop: every stage is event-driven and automatable, and the state model ensures that nothing stale or unapproved reaches your users unless you explicitly ask for it.

## The loop at a glance

| # | Stage | Who drives it | What happens |
|---|---|---|---|
| 1 | **Provision** | Operator (OWNER/ADMIN) | Create the Kalem project, set the base locale, configure settings. |
| 2 | **Setup** | Operator | Add target locales, invite members, create a glossary, configure webhooks. |
| 3 | **Ingest** | Developer / CI | Push new and changed source strings from your codebase into Kalem. |
| 4 | **Enrich** | Developer / Agent | Add context — descriptions, tags, glossary terms — to help translators and AI. |
| 5 | **Translate** | Agent (LLM) / Human editor | Produce translations for each locale. Fills the working `value` slot. |
| 6 | **Approve** | Reviewer (MANAGER/EDITOR) | Sign off on translations. Snapshots `value` into `approvedValue`. |
| 7 | **Deliver** | Developer / CI | Pull approved translations out of Kalem into your app's locale files. |
| ↺ | **Loop** | — | New source changes re-enter at Ingest; the cycle continues. |

## Stage-by-stage breakdown

<Steps>

<Step title="Provision">
An OWNER or ADMIN creates a project and sets its `baseLocale` — the language your developers write strings in. This is a one-time setup step. Everything else in the project is scoped to that base locale.
</Step>

<Step title="Setup">
With a project in place, an OWNER or ADMIN adds the target locales you want to support, invites team members and assigns them project roles, populates the glossary with preferred translations and terms that must not be translated, and optionally configures webhooks to notify external systems when translation events occur.
</Step>

<Step title="Ingest">
Your developers or CI pipeline calls `kalem push` (or the equivalent REST import endpoint) to sync the current set of source strings into Kalem. Kalem compares the incoming keys against what it already holds:

- **New keys** are created with status `untranslated` in every target locale.
- **Changed base values** cause all existing translations for that key to be flagged `stale`.
- **Keys that no longer appear** are marked `deprecated` within their namespace. If a deprecated key reappears in a later push, it is automatically restored to `active`.

<Note>
  Ingest is namespace-scoped. You can push one namespace at a time, which makes it safe to integrate into incremental CI jobs that only touch changed parts of your codebase.
</Note>
</Step>

<Step title="Enrich">
After ingestion, you (or an AI agent) can add context to keys: human-readable `description` fields, `tags` for filtering, and links to glossary terms. This step is optional but significantly improves translation quality, especially when using LLM-based translation. Good descriptions tell a translator where the string appears and what tone is appropriate.
</Step>

<Step title="Translate">
An AI agent with an EDITOR role (or a human translator) writes translations into the working `value` slot. Translations start at status `untranslated` and move to `translated` once a value is written.

The `origin` field is set automatically based on how the translation was produced:

| Origin value | Meaning |
|---|---|
| `llm` | Written by a large language model via the MCP server |
| `human` | Written by a person in the editor UI or via MCP |
| `tm` | Matched from translation memory |
| `mt` | Produced by a machine translation service |
| `import` | Loaded via file import |
</Step>

<Step title="Approve">
A reviewer with EDITOR or MANAGER role inspects the translated strings and approves the ones that meet quality standards. On approval:

1. The translation's `status` moves from `translated` to `approved`.
2. The current `value` is copied into the `approvedValue` snapshot.
3. The `stale` flag is cleared (if it was set).

Approval is the gate between "being worked on" and "ready to ship". Until a translation is approved, it will not appear in the default delivery output.
</Step>

<Step title="Deliver">
Your developer or CI pipeline calls `kalem pull` to export the approved translations as locale files. By default, `kalem pull` outputs the `approvedValue` for every key — the stable, reviewed snapshot.

```bash
# Pull approved translations (default)
kalem pull --project my-app --locale fr

# Pull working drafts instead
kalem pull --project my-app --locale fr --working
```

Use `--working` only in development environments where you want to preview in-progress translations before they are approved.
</Step>

</Steps>

## State model

Understanding how Kalem tracks state helps you predict what happens as strings and translations evolve.

### Key lifecycle: `active` and `deprecated`

A TranslationKey is always in one of two states within its namespace:

- **`active`** — the key is present in the current source. It is pushed regularly and translations are expected to be maintained.
- **`deprecated`** — the key was not present in the most recent push for its namespace. Kalem retains deprecated keys and their translations rather than deleting them.

If a deprecated key reappears in a subsequent push, Kalem restores it to `active` automatically. This means renaming a namespace or temporarily removing a string does not destroy translation history.

<Warning>
  Deprecation is namespace-scoped. If you push namespace `common` and omit a key that exists in namespace `auth`, the `auth` key is not affected. Always push the full contents of each namespace you intend to update.
</Warning>

### Translation status: `untranslated → translated → approved`

Each translation moves forward along this chain:

```
untranslated ──► translated ──► approved
```

Status never moves backward automatically. If an approved translation is edited, it returns to `translated` (not `approved`) until it is reviewed again.

### Dual-slot: `value` and `approvedValue`

Every translation record holds two string slots:

- **`value`** — the working draft. Updated on every write, regardless of who or what produced it.
- **`approvedValue`** — the delivery snapshot. Only updated when the translation is approved.

This separation means you can have active editing and a stable delivery channel running in parallel. Your app always sees the last approved version via `kalem pull`, even while an editor is revising the draft.

### Staleness

When the base-locale `value` for a TranslationKey changes (for example, an English source string is reworded), all existing translations for that key in every target locale are automatically flagged `stale = true`. A stale translation is not blocked from delivery — you can still ship it — but the flag tells reviewers that the translation may no longer match the source.

Approving a stale translation clears the flag and takes a new `approvedValue` snapshot.

### Origin provenance

The `origin` field on every translation records how it was produced. This is read-only metadata — you cannot set it manually. It is useful for filtering translations by quality tier or for auditing how much of your content was human-reviewed versus AI-generated.

## Git as the release layer

Kalem does not have its own concept of releases or versioned snapshots. Your locale files are plain text — commit them to your repository like any other source file. A git tag or a deploy commit is your frozen release. This keeps the delivery model simple: `kalem pull` writes files, your CI commits them, your deploy process picks them up. There are no over-the-air translation objects to manage.
