Every request to Kalem — whether through the MCP server or the REST API — must include an `Authorization: Bearer <api-key>` header. Kalem resolves that key to a user, then evaluates the user's roles before allowing any operation to proceed. There are no anonymous requests. If a key is missing, expired, or revoked, the request receives a `401` response immediately.

## Two role dimensions

Kalem uses two orthogonal role dimensions. You need both to understand what a user can do.

### Global roles

Global roles apply across the entire Kalem organization. Every user has exactly one global role.

| Role | Description |
|---|---|
| **OWNER** | Full control. Can manage all projects, all users, and all global settings. Only an OWNER can grant or change privileged global roles. The last OWNER in an organization cannot be demoted. |
| **ADMIN** | Broad administrative access. Can manage projects and users but cannot change OWNER-level roles. |
| **MEMBER** | Standard access. Can work within projects they have been explicitly added to. Has no org-wide administrative privileges. |

### Per-project roles

Per-project roles are granted through a Membership record that links a user to a specific project. A user can hold different per-project roles on different projects.

| Role | Description |
|---|---|
| **MANAGER** | Full control over the project, including managing members, locales, and keys. Can approve translations. |
| **EDITOR** | Can write and approve translations, manage glossary terms, and add context to keys. Designed for translators and AI agents doing translation work. |
| **DEVELOPER** | Can push and pull locale files (ingest and deliver). Designed for CI pipelines and developer tooling. Cannot write or approve translations. |
| **VIEWER** | Read-only access to all project content. Cannot modify anything. |

## Permission matrix

The table below shows what each role can do. OWNER and ADMIN share the same permission set across all projects.

| Operation | OWNER / ADMIN | MANAGER | EDITOR | DEVELOPER | VIEWER |
|---|:---:|:---:|:---:|:---:|:---:|
| Read project, keys, and translations | ✅ | ✅ | ✅ | ✅ | ✅ |
| Write translations (working `value`) | ✅ | ✅ | ✅ | ❌ | ❌ |
| Approve translations | ✅ | ✅ | ✅ | ❌ | ❌ |
| Manage keys (create, update, deprecate) | ✅ | ✅ | ❌ | ✅ | ❌ |
| Manage locales (add, remove) | ✅ | ✅ | ❌ | ❌ | ❌ |
| Manage project members | ✅ | ✅ | ❌ | ❌ | ❌ |
| Create projects / manage org users | ✅ | ❌ | ❌ | ❌ | ❌ |

<Note>
  DEVELOPER can manage keys (push source strings) but cannot write or approve translations. This is intentional: CI should be able to ingest new source strings without being able to overwrite human or AI translations.
</Note>

## OWNER vs ADMIN

OWNER and ADMIN have nearly identical day-to-day permissions, with one important difference: **only an OWNER can grant or change privileged global roles** (OWNER and ADMIN). An ADMIN cannot promote another user to ADMIN or OWNER, and cannot demote an OWNER.

Additionally, Kalem enforces that at least one OWNER must always exist in the organization. If you are the last OWNER, you cannot demote yourself until another user has been promoted to OWNER.

<Warning>
  Be careful when provisioning the first OWNER of a new Kalem instance. There is no recovery path if you lose access to the sole OWNER account.
</Warning>

## Roles across the lifecycle

Each stage of the [continuous localization loop](/concepts/lifecycle) maps to a role:

| Lifecycle stage | Who drives it | Required role |
|---|---|---|
| Provision, Setup | Operator | Global OWNER or ADMIN |
| Ingest, Deliver (CI) | Developer / pipeline | Project DEVELOPER |
| Translate | AI agent or human translator | Project EDITOR |
| Approve | Reviewer | Project EDITOR or MANAGER |
| Manage project members / locales | Team lead | Project MANAGER or global ADMIN/OWNER |

This mapping makes it straightforward to set up least-privilege access: give your CI service account a DEVELOPER key, give your translators (human or AI) an EDITOR key, and give your team leads a MANAGER membership.

## API key scope and expiry

An API key is always tied to a specific user and inherits that user's roles. However, you can further restrict a key at creation time using two optional constraints:

### `readOnly`

Setting `readOnly: true` on a key means that key can only perform read operations, regardless of what roles the underlying user holds. Use this for keys you share with external monitoring tools, dashboards, or read-only integrations where you want an extra layer of protection against accidental writes.

### `expiresAt`

You can set an ISO-8601 expiry timestamp on a key. Once that instant passes, the key returns a `401` on every request. Expired keys are not automatically deleted — you can see them in the key list — but they cannot be reactivated. Create a new key if you need to restore access.

<Note>
  Both `readOnly` and `expiresAt` are set only at creation time and cannot be changed after the fact. If you need to change either property, revoke the old key and issue a new one.
</Note>

## MCP tool scoping

When your AI agent calls `tools/list` on the MCP server, Kalem automatically filters the returned tool catalog to only include the tools the authenticated key's user is permitted to call. An EDITOR key will not see tools that require MANAGER or DEVELOPER permissions, and a `readOnly` key will only see read tools. This means your agent's available actions are always consistent with the key's permissions — there is no risk of an agent attempting an operation it is not allowed to perform.

<Tip>
  Give your CI pipeline a DEVELOPER-scoped API key and your AI translation agent an EDITOR-scoped key. This way, a misconfigured pipeline can never accidentally overwrite translations, and an agent can never push or pull raw locale files outside the intended workflow.
</Tip>
