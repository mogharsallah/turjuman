An AI agent drives Turjuman by connecting to the [MCP](/reference/mcp-tools) with a bearer API key, discovering the operations it needs, and running them. The MCP is a single **code surface**: rather than advertising one tool per capability, it exposes three — `search`, `describe`, and `run_code` — and the agent writes short programs against the operation catalogue. This page explains the interaction model; for step-by-step usage, see [Drive Turjuman over MCP](/guides/code-mode) and [Translate with MCP](/guides/translate-with-mcp).

## The interaction loop

Every agent session follows the same shape:

1. **Connect** — the agent opens a Streamable HTTP connection to the MCP Function URL, passing `Authorization: Bearer <api-key>` on every request. The connection is stateless: there is no session to manage, and the agent can retry or reconnect freely.
2. **Discover** — the agent calls `search` to find the operations (and docs) it needs, then `describe` to confirm exact arguments.
3. **Act** — the agent writes a TypeScript program that calls `await turjuman.<operation>(args)` and runs it with `run_code`. Each call is dispatched to the shared service layer, which enforces authorization before anything runs.

Because the server holds no per-connection state, the agent's only durable context is what Turjuman stores and what the agent keeps in its own reasoning.

## The translate-and-judge loop

The operations exist to serve one loop. An agent typically:

1. **Defines or reads keys.** A developer (or an import) adds a `TranslationKey` — the source string plus whatever context is cheap to attach. The agent reads the keys that need a locale with `list_untranslated` or `list_stale`.
2. **Translates.** The agent writes a draft `value` with `set_translation` (or `bulk_set_translations`). The cell becomes `proposed`.
3. **Judges its own output** and picks an exit: **accept** (confident — call `accept_translation`), **re-loop** (unsure — write a better draft with the objection as new context), or **escalate** (an irreducible judgment a human should make).
4. **Records.** An accept appends an immutable version and advances the cell's `head`. History holds every accepted value, non-destructively.

The human is the *terminal* exit, not a line-by-line reviewer. There is no quality score to submit — the cell's [lifecycle state](/concepts/lifecycle) is the entire verdict. A project that wants a person to sign off sets `requireHumanAccept: true`, which stops a run from accepting its own work.

## Discovery spans operations *and* docs

`search` queries one knowledge index that covers both the **operations** (returned with full typed signatures) and the **documentation** — these guides, concepts, and reference pages. So an agent can learn *how the system works*, not merely which functions exist: a search for "webhook events" surfaces both the relevant operations and the concept page that explains them. This is why the docs are treated as a first-class agent surface, not just a human one.

## Authorization is enforced in the core

The surface never changes the *permissions*. Whether an agent calls an operation directly or from inside a `run_code` sandbox, the request is dispatched to the same handler in the shared service layer with the same authenticated context. A read-only key still can't write — the write is denied in the core, not merely hidden.

The key an agent presents determines what it can do: a **read-only key** can only invoke read operations, and a key with global **MEMBER** role cannot invoke org-level administration operations. See [Roles & permissions](/concepts/roles-and-permissions) for the underlying model.

## Common misconceptions

<Note>
  **"`run_code` lets the agent run arbitrary code on my server."** It runs the agent's TypeScript in a sandbox whose only capability is the Turjuman SDK — no network, filesystem, environment, or access to the discovery tools. Every operation call still passes through the same authorization as any other request.
</Note>

<Note>
  **"The agent submits a quality score."** It does not. Quality is a lifecycle state, not a number — the agent accepts, re-loops, or escalates, and a stored score would only go stale. See [Lifecycle](/concepts/lifecycle).
</Note>
