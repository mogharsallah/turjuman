Turjuman exposes two surfaces for working with your translation data: an **MCP** designed for AI agents and agentic IDEs like Claude Code, and a **REST API** designed for deterministic CLI and CI workflows. Both surfaces are thin transport adapters ‚Äî they translate incoming requests into calls against the same shared service layer, which owns all business logic, authorization, and persistence. The two surfaces cannot drift out of sync because neither contains logic of its own.

Every inbound request ‚Äî an MCP tool call or a REST request ‚Äî carries a `Bearer` API key in the `Authorization` header. The service layer resolves that key to a user and their roles before any operation runs.

The infrastructure is entirely serverless: Turjuman runs on **AWS Lambda Function URLs** (no API Gateway in the path) backed by a **single DynamoDB table**. A `/health` endpoint is available for uptime checks and deployment verification.

## Two surfaces, one brain

Between the transports and the service layer sits a single **operation layer**: every capability is declared once ‚Äî its typed inputs and outputs, its description, and a handler that calls the service. The MCP and the REST API are both projections of those same declarations, so a capability behaves and authorizes identically no matter which surface reaches it.

The transports are deliberately thin. All business logic ‚Äî RBAC enforcement, lifecycle transitions, staleness detection, glossary lookups ‚Äî lives exclusively in the shared service layer. This means:

- A bug fix or a new permission rule lands in one place and takes effect across both surfaces at once.
- Adding a capability never risks introducing a parallel code path with different authorization behavior.
- You can call either surface and get the same result for the same operation.

<Note>
  If you are integrating Turjuman into an AI agent workflow, use the MCP ‚Äî it exposes the full operation catalogue. Use the REST API when you need file-shaped, deterministic operations in a script or CI pipeline.
</Note>

## The MCP is a code surface

The MCP does not advertise one tool per operation. It exposes exactly three tools and asks the agent to write code against the catalogue:

- `search` ‚Äî find what exists, across both the operations (returned with typed signatures) and the documentation (guides, concepts, reference), so an agent learns *how the system works*, not just which functions exist.
- `describe` ‚Äî return the full detail for one result: an operation's typed inputs, output, and hints, or a whole docs page.
- `run_code` ‚Äî run TypeScript that calls `await turjuman.<operation>(args)` and returns a value.

The code runs in an isolated **QuickJS-WASM sandbox** whose only capability is the operation layer ‚Äî no network, filesystem, or environment access, and the discovery tools are not reachable from inside it. Each `turjuman.<operation>(args)` call is dispatched in-process to the very same handler a normal request runs, with the request's authenticated context, so authorization is unchanged; only the final result returns.

This collapses a multi-step job into a single round-trip and keeps the operation catalogue out of the model's context, a large token saving on a big toolset. See the [MCP guide](/guides/code-mode) for how to drive it.

## Why the REST surface is narrow

The REST API is intentionally scoped to a small set of **file-shaped, automatable operations**: listing projects and locales, exporting translation files, and importing them back. These are the actions a CI job or a developer script needs to run reliably without interactive decision-making.

Everything else ‚Äî creating and managing projects, writing and accepting translations, managing glossary terms, configuring webhooks ‚Äî is reached through the MCP. This keeps the REST API predictable and easy to script, while giving agents a rich, expressive surface.

## Domain model

These are the core entities you interact with when using Turjuman.

| Entity | Key fields | Notes |
|---|---|---|
| **Project** | `id`, `name`, `slug`, `baseLocale`, `contextRevision`, `requireHumanAccept` | The top-level container. `baseLocale` is the source language. `requireHumanAccept` gates whether a run may accept its own work. |
| **Branch** | `id`, `name`, `parentBranchId`, `forkPoint`, `status` | A copy-on-write line of work. `main` always exists with `parentBranchId = null`. |
| **Namespace** | `id`, `projectId`, `name`, `lifecycle` | An optional grouping of keys by feature-area. An opaque-id entity; `name` is a renamable label. |
| **Locale** | `code`, `projectId`, `lifecycle` | A target language added to a project (e.g. `fr`, `ja`). |
| **TranslationKey** | `id`, `namespaceId`, `name`, `description`, `tags`, `plural`, `state`, `sourceRevision` | Identifies a single string. `id` is stable identity; `(namespaceId, name)` are renamable labels. `state` is `active` or `deprecated`. |
| **Translation** (cell) | `branchId`, `keyId`, `locale`, `value`, `head`, `lifecycle`, `stale`, `sourceRef`, `origin` | The living draft for a key in one locale. See [Lifecycle](/concepts/lifecycle). |
| **TranslationVersion** | `keyId`, `locale`, `seq`, `value`, `acceptedBy` \| `runRef`, `prevVersionRef` | An immutable accepted-value commit. The cell's `head` points at the current one. |
| **TranslationRun** | `id`, `branchId`, `trigger`, `valueSource`, `status`, `budgetSpent` | The write primitive that records what an agent did across a set of cells. A merge is a run with `trigger = merge`. |
| **Release** | `id`, `branchId`, `label`, `locales`, `status`, `entries` | An immutable shipped snapshot. Pins one branch's accepted view; each entry is `{ keyId, locale, versionRef }`. See [Branching & releases](/concepts/branching-and-releases). |
| **FieldReport** | `id`, `branchId`, `keyId`, `locale`, `releaseRef`, `status`, `resolution` | Production feedback that a shipped string is wrong; filing reopens the cell (`accepted ‚Üí proposed`). |
| **Glossary term** | `term`, `translations`, `doNotTranslate`, `caseSensitive` | A preferred translation or do-not-translate entry that guides AI translation. |
| **Webhook** | `url`, `events`, `projectId` | An outbound notification sent when translation events occur. |
| **User** | `id`, `email`, `globalRole` | A person or service account with a global org role. |
| **Membership** | `userId`, `projectId`, `role` | Grants a user a role on a specific project. |
| **API key** | `id`, `userId`, `readOnly`, `expiresAt` | A bearer token scoped to a user. `readOnly` and `expiresAt` are set at creation and cannot be changed. |

### Keys carry a stable identity

A key's identity is its opaque `id`; its `(namespace, name)` are *display labels*, not identity. Renaming or moving a key is a small metadata change to the key's own row ‚Äî and because every translation cell and version references the `keyId`, **a rename moves no translation data**. Imports and file exports still address keys by name through a per-branch `name ‚Üí id` lookup, so the label stays ergonomic while the identity stays stable. This is why `rename_key` preserves a key's entire translation history.

### The cell and its version chain

A **Translation** is the living cell for one `(branchId, keyId, locale)`: a mutable working `value`, a `lifecycle` state, and a `head` pointer. Accepting a value appends an immutable **TranslationVersion** and advances `head` ‚Äî accepted text lives in that append-only chain, never in a mutable field. This separation of the *living* draft from the *historical* accepted values is what makes acceptance non-destructive and gives every string a full, auditable history. The mechanics ‚Äî the five states, the head compare-and-swap, staleness ‚Äî are covered in [Lifecycle](/concepts/lifecycle).

## Single-table layout

Turjuman stores everything in one DynamoDB table with three global secondary indexes (GSI1‚ÄìGSI3). Records are grouped into three partition families plus the identity singletons:

| Partition (PK) | Sort key (SK) | Holds |
|---|---|---|
| `USER#<id>`, `USEREMAIL#<email>`, `APIKEY#<hash>` | ‚Äî | Identity and access singletons (GSI1 `ORG#<orgId>` lists them per org). |
| `PROJECT#<id>` | `LOCALE#`, `MEMBER#`, `GLOSSARY#`, `WEBHOOK#`, `QACONFIG`, `BRANCH#`, `NS#`, `RUN#` | The project record and its singletons: locales, members, glossary, webhooks, QA config, branches, namespaces, runs. |
| `PROJ#<id>#BR#<branch>` | `KEY#<keyId>`, `KEYNAME#<ns>#<name>` | A branch's key definitions and the `name ‚Üí id` lookup rows. |
| `PROJ#<id>#BR#<branch>#LOC#<code>` | `KEY#<keyId>`, `VER#<keyId>#<seq>` | A branch √ó locale's live cells and their append-only version chains. |
| `PROJ#<id>#REL#<releaseId>` | `META`, `KEY#<keyId>#<code>` | An immutable release: a metadata row plus one pinned `{ keyId, locale, versionRef }` entry per shipped cell. |

Three design choices fall out of this layout:

- **Branching is one PK segment.** `#BR#<branchId>` scopes both key definitions and cells. `main` is not special ‚Äî it is `branchId = "main"` baked into the key. A copy-on-write read is at most a few point-reads up the parent chain; a branch that overrode nothing collapses to a single fall-through read of its parent.
- **A release is its own immutable partition.** `PROJ#<id>#REL#<releaseId>` holds a metadata row plus the pinned `{ keyId, locale, versionRef }` entries ‚Äî a self-contained snapshot of what shipped, so listing releases and reassembling one never touch the live cells. A [merge](/concepts/branching-and-releases) needs no new storage: it is a run that overlays one branch's cells onto another.
- **`GSI3` flips locale-partitioned cells to key-major.** Indexed by `PROJ#<id>#BR#<branch>#KEY#<keyId>` ‚Üí `LOC#<code>` and set on live cells only, it reads one key across every locale without scanning. Because cells are keyed by `keyId`, this index is rename-safe.

Read [the lifecycle](/concepts/lifecycle) for how cells move through their states, and [Roles & permissions](/concepts/roles-and-permissions) for how the service layer authorizes each call.
