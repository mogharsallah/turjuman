Turjuman exposes two surfaces for interacting with your translation data: an **MCP server** designed for AI agents and agentic IDEs like Claude Code, and a **REST API** designed for deterministic CLI and CI workflows. Both surfaces are thin transport adapters — they translate incoming requests into calls against the same shared service layer, which owns all business logic, authorization, and persistence. There is no way for the two surfaces to drift out of sync because neither contains logic of its own.

## Request flow

```
Claude Code / agent ──Streamable HTTP + Bearer key──► MCP Server ─┐
Developer CLI / CI ──REST + Bearer key──────────────► REST API ───┤
                                                                   └─► Service layer → DynamoDB
```

Every inbound request — whether it arrives as an MCP tool call or a REST request — carries a `Bearer` API key in the `Authorization` header. The service layer resolves that key to a user and their roles before any operation runs.

The infrastructure is entirely serverless: Turjuman runs on **AWS Lambda Function URLs** (no API Gateway in the path) backed by a **single DynamoDB table**. A `/health` endpoint is available for uptime checks and deployment verification.

## Two surfaces, one brain

Between the transports and the service layer sits a single **operation layer**: every capability is declared once — its typed inputs and outputs, its description, and a handler that calls the service. The MCP server projects each operation as a tool, the REST API projects the applicable ones as routes, and code mode (below) exposes them to a sandbox. Because all three are projections of the same declarations, a capability behaves and authorizes identically no matter which surface reaches it.

The MCP server and the REST API are deliberately kept as thin as possible. All business logic — including RBAC enforcement, translation state transitions, staleness detection, and glossary lookups — lives exclusively in the shared service layer. This means:

- A bug fix or a new permission rule lands in one place and takes effect across both surfaces simultaneously.
- Adding a new MCP tool never risks introducing a parallel code path with different authorization behavior.
- You can call either surface interchangeably and always get the same result for the same operation.

<Note>
  If you are integrating Turjuman into an AI agent workflow, prefer the MCP server — it gives your agent access to the full 40+ tool catalog. Use the REST API when you need file-shaped, deterministic operations in a script or CI pipeline.
</Note>

## Why the REST surface is narrow

The REST API is intentionally scoped to a small set of **file-shaped, automatable operations**: listing projects and locales, exporting translation files, and importing them back. These are the actions a CI job or a developer script needs to run reliably without any interactive decision-making.

Everything else — creating and managing projects, inviting members, writing translations, managing glossary terms, configuring webhooks — is exposed exclusively through the MCP server. This keeps the REST API predictable and easy to integrate into shell scripts and pipelines, while giving agents a rich, expressive surface to work with.

## Code mode and the sandbox

The MCP server can run in **code mode** (`?mode=code`), an alternative to advertising the full tool catalogue. Instead of one tool per operation, it exposes two: `search_sdk` to discover operations and `run_code` to execute TypeScript that calls them. The code runs in an isolated **QuickJS-WASM sandbox** whose only capability is the operation layer — there is no network, filesystem, or environment access. Each `turjuman.<operation>(args)` call is dispatched in-process to the very same handler a normal MCP tool call runs, with the request's authenticated context, so authorization is unchanged; only the final result returns.

This collapses a multi-step job into a single round-trip and keeps the tool catalogue out of the model's context, which is a large token saving on big toolsets. The two modes are mutually exclusive per connection. See the [Code mode guide](/guides/code-mode) for details.

## Domain model

These are the core entities you interact with when using Turjuman.

| Entity | Key fields | Notes |
|---|---|---|
| **Project** | `id`, `name`, `slug`, `baseLocale` | The top-level container. `baseLocale` is the source language (e.g. `en`). |
| **Locale** | `code`, `projectId` | A target language added to a project (e.g. `fr`, `ja`). |
| **TranslationKey** | `namespace`, `name`, `description`, `tags`, `plural`, `state` | Identifies a single string within a project. Namespaces group related keys together. `state` is `active` or `deprecated`. |
| **Translation** | `value`, `approvedValue`, `status`, `stale`, `origin` | The actual string data for a key in a specific locale. See below. |
| **Glossary term** | `term`, `translations`, `doNotTranslate`, `projectId` | A preferred translation or do-not-translate entry used to guide AI translation. |
| **Webhook** | `url`, `events`, `projectId` | An outbound notification sent when translation events occur. |
| **User** | `id`, `email`, `globalRole` | A person or service account with a global org role. |
| **Membership** | `userId`, `projectId`, `role` | Grants a user a role on a specific project. |
| **API key** | `id`, `userId`, `readOnly`, `expiresAt` | A bearer token scoped to a user. `readOnly` and `expiresAt` are set at creation and cannot be changed. |

### The Translation entity in detail

A Translation holds two string slots:

- **`value`** — the working draft. This is updated whenever a translation is written, whether by a human editor, an AI agent, or an import.
- **`approvedValue`** — a stable snapshot of the last approved version. This is only updated when a translation moves to the `approved` status.

The `status` field follows a linear progression: `untranslated` → `translated` → `approved`. A translation can also carry a `stale` flag, which is set automatically when the base-locale value for that key changes — signaling that the translation may no longer be accurate.

The `origin` field records how a translation was produced: `human`, `llm`, `tm` (translation memory), `mt` (machine translation), or `import`.

<Tip>
  Because `approvedValue` is a stable snapshot, you can always deliver approved translations to your app even while editors are actively revising the working `value`. The two slots let review workflows run in parallel with delivery.
</Tip>
