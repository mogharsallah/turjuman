Turjuman adds a **governed AI review pass** on top of the deterministic [QA checks](/guides/quality-checks):
your connected agent grades each translation **0–100** against the **MQM** (Multidimensional Quality
Metrics) rubric, and the server **auto-routes** the result — a high score can auto-approve, a low score
is flagged for a human. It's the *review* half of stage 5 in the [lifecycle](/concepts/lifecycle).

Turjuman is **BYO-LLM and serverless — the server never runs a model.** Instead it does three things:

1. **Serves the methodology.** It assembles the grading prompt (the MQM rubric + your project's
   guidance + glossary + the source/target strings) so every client scores the same way.
2. **Records + routes.** It stores the score with provenance and moves the translation between
   `translated`, `needs_review`, and `approved`.
3. **Stays out of the model's way.** The grading runs inside whatever agent you connect (Claude Code,
   opencode, a CI agent…), with your keys.

<Note>
  **Score honestly, and avoid self-grading bias.** A model grading its *own* output tends to over-rate
  it. Run scoring as a **separate review pass** — ideally with a different API key and a different model
  from the one that produced the translation. The `review_translations` flow is built for exactly this.
</Note>

## How a score routes

After you submit a score, the server routes the translation using the per-project config:

| Condition | Result |
|---|---|
| `score < threshold` | `needs_review` (held back for a human) |
| `score ≥ threshold`, **auto-approve on**, key holds `translation.review`, value is machine-origin | `approved` (promotes `value → approvedValue`) |
| `score ≥ threshold` but any of the above is missing | `translated` (score recorded, left for a human) |

Two guardrails are deliberate: **auto-approve is opt-in (off by default)**, and it **never fires on
`origin: "human"` work** — a person's own edit is never auto-approved by a score. The score augments
the role gate; it doesn't replace it.

You can only score a translation that already **has a value** — scoring an untranslated or empty key is
rejected (single submit) or skipped (batch). The base locale is the source of truth and can't be scored
against itself.

## Configure scoring

Set the bar and (optionally) project-specific evaluation guidance once. The `guidance` text is merged
into the grading prompt, and every score it produces is stamped with a `+custom` prompt version for
auditability.

<Tabs>
  <Tab title="MCP (agent)">
    > "Set the scoring config: threshold 90, turn on auto-approve, and add guidance to prefer the
    > formal register (vous, not tu) for French."

    `set_score_config { projectId, threshold?, autoApprove?, guidance? }` — requires a
    project-management role. `get_score_config` reads it back. Defaults are `threshold: 90`,
    `autoApprove: false`.
  </Tab>
  <Tab title="REST">
    ```bash
    curl -X PUT https://<api-url>/v1/projects/<id>/score-config \
      -H "Authorization: Bearer op_live_..." \
      -H "Content-Type: application/json" \
      -d '{ "threshold": 90, "autoApprove": true, "guidance": "Prefer the formal register (vous)." }'
    ```
  </Tab>
</Tabs>

## Score in a review pass

<Steps>
  <Step title="Fetch the grading prompt">
    Over MCP, get the server-assembled prompt (the `prompts` capability): `score_translation` for one
    key, or `review_locale` for a page of a locale. It contains the MQM rubric, your guidance, the
    glossary, and the source/target strings, ending with a strict JSON output contract.

    Over REST, `GET /v1/projects/:id/translations/score-prompt?locale=fr&name=greeting` returns the
    same assembled prompt (omit `name` for the batch form).
  </Step>
  <Step title="Grade with the model">
    The agent's model returns `{ "score": 0-100, "comment": "…" }` (or a JSON array for the batch
    form), citing the main MQM issues.
  </Step>
  <Step title="Submit the score">
    `score_translation { projectId, locale, name, namespace?, score, comment?, model? }` for one, or
    `review_translations { projectId, locale, entries[] }` for a batch. Pass `model` so the score
    records which model produced it. The server routes per the table above and returns the updated
    translation (or, for a batch, `{ written, skipped[], approved, flagged }`).
  </Step>
</Steps>

A typical agent instruction ties it together:

> "Using a separate review pass, fetch the `review_locale` prompt for `fr`, grade each translation
> against the MQM rubric, and submit the scores with `review_translations`. Then list anything that
> landed in `needs_review`."

## Work the review queue

Low-scored translations land in `needs_review` — they carry a value but are **not** delivered (only
`approvedValue` ships). Surface them, then fix or approve:

<Tabs>
  <Tab title="MCP (agent)">
    `list_for_review { projectId, locale }` returns the flagged keys (paged). For each, the agent can
    read the `score`/`scoreComment`, edit the `value` (back to `translated`), or — with a review role —
    approve it via `set_translation_status`.
  </Tab>
  <Tab title="REST">
    ```bash
    curl https://<api-url>/v1/projects/<id>/translations/for-review?locale=fr \
      -H "Authorization: Bearer op_live_..."
    ```
  </Tab>
</Tabs>

## CI sketch

Because the whole flow is BYO-LLM, you can run it headless: a scheduled job points a (review-role) key
and a *different* model at the `review_locale` prompt, submits the scores, and lets the threshold
auto-approve the confident ones while everything else waits in `needs_review` for a human. Nothing
ships until it's `approved`, so an over-eager score can never reach production.

## Where it fits

- **Scale & severities:** 0–100, graded across the seven MQM-Core dimensions (terminology, accuracy,
  linguistic conventions, style, locale conventions, audience appropriateness, design & markup) with
  minor/major/critical severities. Turjuman doesn't compute the MQM penalty formula — the model emits
  the 0–100 score directly.
- **One number + a comment now; a per-category scorecard later.** v1 stores a single score plus the
  model's rationale (`scoreComment`); a per-dimension breakdown is a planned upgrade.
- **Provenance:** every score records `scoredBy`, `scoredAt`, `scoreModel`, and `promptVersion`. A
  re-score overwrites the previous one (no score history).
