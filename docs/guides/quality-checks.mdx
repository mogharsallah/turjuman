QA in Kalem is advisory. It surfaces problems and reports them — it never auto-approves translations or silently discards findings. Use QA output as structured input for human reviewers and as a hard gate in your CI pipeline.

## Running checks

<Tabs>
  <Tab title="MCP">
    Call `run_qa_checks` from your agent:

    ```
    run_qa_checks {
      projectId: "proj_123",
      locale: "fr",          // optional — omit to check all locales
      checks: ["icu", "placeholders"],  // optional — omit to run all enabled checks
      slot: "approved"       // optional — "approved" (default) or "working"
    }
    ```

    The tool returns a list of findings, each with a `checkId`, `severity`, `keyName`, `locale`, and a human-readable `message`.
  </Tab>
  <Tab title="CLI">
    Run checks against a specific locale:

    ```bash
    turjuman check --locale fr
    ```

    Run checks against approved translations only:

    ```bash
    turjuman check --approved
    ```

    Run checks as part of a push:

    ```bash
    turjuman push --check
    ```

    `--check` appended to `push` runs the full check suite after the push completes and exits non-zero if any error-severity findings are reported.
  </Tab>
  <Tab title="REST">
    ```http
    POST /v1/projects/<id>/checks
    Content-Type: application/json
    Authorization: Bearer <your-api-key>

    {
      "locale": "fr",
      "checks": ["icu", "placeholders", "glossary"],
      "slot": "approved"
    }
    ```

    All body fields are optional. Omit `locale` to check all locales; omit `checks` to run all enabled checks.
  </Tab>
</Tabs>

## What Kalem checks

| Check | What it looks for |
|---|---|
| **ICU syntax** | Validates ICU MessageFormat syntax — catches malformed `{variable}`, `{count, plural, ...}`, and `{gender, select, ...}` expressions |
| **Placeholders** | Verifies that every placeholder present in the source string also appears in the translation, with no extras introduced |
| **Plural forms** | Confirms the correct plural categories are covered for the target locale's CLDR plural rules |
| **Markup** | Detects unmatched or mismatched HTML/XML tags |
| **Length** | Flags translations that are significantly shorter or longer than the source, using configurable ratio thresholds |
| **Whitespace** | Catches leading/trailing whitespace mismatches and inconsistent internal spacing |
| **Punctuation** | Checks that terminal punctuation (period, question mark, exclamation) matches the source |
| **Empty** | Reports keys that have no translation value in the target locale |
| **Glossary** | Verifies that glossary terms are translated using their designated equivalents |
| **Duplicates** | Identifies keys whose translation is identical to another key's translation, which may indicate copy-paste errors |
| **Staleness** | Surfaces translations that were marked stale after their source string changed |
| **Coverage** | Reports the percentage of keys that have translations in each locale |

<Note>
  Coverage checking is disabled by default. Enable it in your project's QA config if you want CI to fail on incomplete locales.
</Note>

## Configuring checks per project

Each project has its own QA config. You can enable or disable individual checks and set their severity independently.

<Tabs>
  <Tab title="MCP">
    Read the current config:

    ```
    get_qa_config { projectId: "proj_123" }
    ```

    Update it:

    ```
    set_qa_config {
      projectId: "proj_123",
      config: {
        checks: {
          coverage: { enabled: true, severity: "error" },
          length:   { enabled: true, severity: "warning" }
        }
      }
    }
    ```
  </Tab>
  <Tab title="REST">
    Read the current config:

    ```http
    GET /v1/projects/:id/qa-config
    Authorization: Bearer <your-api-key>
    ```

    Update it:

    ```http
    PUT /v1/projects/:id/qa-config
    Content-Type: application/json
    Authorization: Bearer <your-api-key>

    {
      "checks": {
        "coverage": { "enabled": true, "severity": "error" },
        "length":   { "enabled": true, "severity": "warning" }
      }
    }
    ```
  </Tab>
</Tabs>

Each check entry accepts two optional fields:

| Field | Values | Effect |
|---|---|---|
| `enabled` | `true` / `false` | Include or exclude the check from runs |
| `severity` | `"error"` / `"warning"` / `"info"` | Controls how CI treats the finding — only `"error"` causes a non-zero exit |

### Ignore rules

Suppress specific findings without disabling the entire check by adding ignore rules to your QA config. Rules can match on any combination of `checkId`, `namespace`, `keyName`, and `locale`:

```json
{
  "ignoreRules": [
    { "checkId": "length", "keyName": "legal.disclaimer" },
    { "checkId": "punctuation", "locale": "ja" }
  ]
}
```

<Note>
  QA is advisory by design. Findings are surfaced to reviewers and block CI when configured as errors, but they never gate approval automatically. Promoting a translation to approved is always a role-gated human action.
</Note>
