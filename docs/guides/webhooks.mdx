Webhooks let you react to changes in Turjuman as they happen — new keys appearing, translations being updated, base strings changing and marking downstream translations as stale. Instead of polling the API, register an endpoint and Turjuman will push a signed payload to it immediately after each qualifying event.

## Register a webhook

Use the `add_webhook` MCP tool to register an endpoint:

```
add_webhook {
  projectId: "proj_123",
  url: "https://your-app.example.com/webhooks/turjuman",
  events: ["translation.updated", "key.created", "translation.stale"]
}
```

Turjuman generates a signing secret at registration time. **Copy it immediately** — it is shown only once and cannot be retrieved again. Store it in your secrets manager.

Manage your webhooks with:
- `list_webhooks { projectId }` — returns all registered endpoints and their subscribed events
- `remove_webhook { projectId, webhookId }` — unregisters an endpoint

## Events

| Event | Fired when |
|---|---|
| `translation.updated` | A translation's value or status changes |
| `translation.stale` | A base value changes and dependent translations are flagged stale |
| `key.created` | A new key is added to the project |
| `key.updated` | A key's metadata (name, namespace, description) is edited |
| `key.deleted` | A key is deleted from the project |
| `locale.added` | A new locale is added to the project |

## Payload format

Every event delivers a JSON payload with the same envelope:

```json
{
  "event": "translation.updated",
  "projectId": "proj_x",
  "timestamp": "2024-01-15T10:30:00Z",
  "data": { "key": "greeting", "locale": "fr", "status": "translated" }
}
```

The shape of `data` varies by event type but always contains the identifiers relevant to what changed.

## Request headers

Each webhook delivery includes two custom headers:

| Header | Value |
|---|---|
| `X-Turjuman-Event` | The event name, e.g. `translation.updated` |
| `X-Turjuman-Signature` | `sha256=<hex>` — an HMAC-SHA256 signature of the raw request body |

## Verify the signature

Always verify the signature before processing a webhook payload. Compute the expected HMAC over the **raw request body** using your signing secret and compare it to the `X-Turjuman-Signature` header using a timing-safe comparison:

```js
import crypto from "node:crypto";

const expected = "sha256=" + crypto
  .createHmac("sha256", secret)
  .update(rawBody)
  .digest("hex");

const ok = crypto.timingSafeEqual(
  Buffer.from(expected),
  Buffer.from(signatureHeader)
);

if (!ok) {
  return res.status(401).send("Invalid signature");
}
```

<Warning>
  Compute the HMAC over the **raw request body bytes** — before any JSON parsing. Parsing and re-serializing the body may alter whitespace or key ordering, producing a different byte sequence and causing valid signatures to fail verification.
</Warning>

## Delivery guarantees

Turjuman delivers webhooks on a **best-effort, at-most-once** basis:

- **No retries.** If your endpoint doesn't respond with a `2xx` status within 5 seconds, the delivery is dropped. Failed deliveries do not block events to other registered endpoints.
- **No ordering guarantees.** Events for the same resource may arrive out of order under load.
- **Deduplication is your responsibility.** Use the `timestamp` and resource identifiers in the payload to detect and discard duplicates.

<Note>
  Treat webhook events as hints, not as a reliable audit log. An event tells you that something changed — not necessarily everything that changed. Make your handlers idempotent and always re-fetch the current state from the API when you need authoritative data.
</Note>
