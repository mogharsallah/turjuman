Self-hosting Kalem means running it in your own AWS account: one DynamoDB table and up to three Lambda Function URLs (MCP server, REST API, and webhook dispatcher). The `@turjuman/aws-deploy` package handles everything — CDK synthesis, bootstrapping, CloudFormation deployment, and first-owner creation — in a single interactive command. You don't need the SAM CLI, a separate CDK bootstrap step, or a clone of the Kalem repo.

<Warning>
  The CDK stack is conventional but has **not yet been verified end-to-end against a live AWS account**. Use the [Quickstart](/quickstart) to evaluate Kalem locally first, and treat the AWS deploy as beta. Please report any issues you encounter.
</Warning>

## Prerequisites

- An AWS account with credentials configured (via environment variables, `~/.aws/credentials`, or an IAM role)
- Node.js 20 or later
- Sufficient IAM permissions to create CloudFormation stacks, Lambda functions, DynamoDB tables, and IAM roles

## Deploy

<Steps>
  <Step title="Run the deploy command">
    Run the interactive deployer with `npx` — it ships as a self-contained package with pre-bundled Lambda assets, so no repo clone is required:

    ```bash
    npx @turjuman/aws-deploy deploy
    ```

    The command prompts you for:
    - CloudFormation stack name (default: `kalem`)
    - AWS region
    - Allowed CORS origins
    - Lambda memory (MB) and timeout (seconds)
    - Whether to deploy the REST API surface and the webhook dispatcher
    - DynamoDB point-in-time recovery and deletion protection settings
    - Whether to run functions inside a VPC (optional)
    - First owner email and name

    After the CloudFormation stack reaches `CREATE_COMPLETE` (or `UPDATE_COMPLETE`), the command prints your stack outputs and API key:

    ```
    ┌ Stack outputs ──────────────────────────────────────┐
    │ MCP URL:   https://xxxx.lambda-url.us-east-1.on.aws │
    │ API URL:   https://yyyy.lambda-url.us-east-1.on.aws │
    │ Table:     turjuman-TurjumanTable-XXXXXXXXXXXX        │
    └─────────────────────────────────────────────────────┘

    ┌ First owner ────────────────────────────────────────┐
    │ API key: km_live_xxxxxxxxxxxxxxxxxxxx               │
    │                                                     │
    │ Store this now — it is shown only once.             │
    │ Saved to ~/.turjuman/auth.json (you're logged in).  │
    └─────────────────────────────────────────────────────┘
    ```

    <Note>
      Your API key is shown **once** and is not stored in plaintext. Copy it to a secrets manager or password vault immediately. If the org already has users (e.g. on a re-deploy), the command skips owner creation and tells you to reuse your existing key.
    </Note>
  </Step>

  <Step title="Save your deploy config">
    The deployer writes a `turjuman.deploy.json` file in your working directory. This file caches your settings (stack name, region, Lambda tuning, etc.) so subsequent re-deploys are non-interactive:

    ```bash
    npx @turjuman/aws-deploy deploy   # reads turjuman.deploy.json; no prompts
    ```

    Use `--reconfigure` to re-prompt for all settings:

    ```bash
    npx @turjuman/aws-deploy deploy --reconfigure
    ```

    `turjuman.deploy.json` contains no secrets — it is safe to commit. The canonical copy of your config is also stored in AWS SSM Parameter Store so you can redeploy from any machine.
  </Step>
</Steps>

## Stack outputs

| Output | Description |
|--------|-------------|
| `McpUrl` | The Lambda Function URL for the MCP server. Use this in your `.mcp.json`. |
| `ApiUrl` | The Lambda Function URL for the REST API. Use this with `turjuman login`. |
| `TableName` | The DynamoDB table name. Needed for the `bootstrap` command and manual operations. |

## Deploy options

You can override individual settings non-interactively using `--set`, `--enable`, and `--disable`:

| Option | Example | Description |
|--------|---------|-------------|
| `--set` | `--set mcp.memorySize=512` | Set a specific config value (`path=value`). Repeatable. |
| `--enable` | `--enable webhook` | Enable a surface (`api` or `webhook`). |
| `--disable` | `--disable webhook` | Disable a surface (`api` or `webhook`). |
| `--skip-bootstrap` | | Skip the CDK bootstrap step (for already-bootstrapped accounts). |
| `--reconfigure` | | Re-prompt for all settings instead of reusing saved config. |

**Common tuning knobs via `--set`:**

| Path | Default | Description |
|------|---------|-------------|
| `functionDefaults.memorySize` | `256` | Lambda memory (MB) applied to all functions. |
| `functionDefaults.timeout` | `15` | Lambda timeout (seconds) applied to all functions. |
| `mcp.memorySize` | — | Override memory for the MCP function only. |
| `corsAllowOrigins` | `["*"]` | Comma-separated allowed origins for Function URL CORS. |
| `table.pointInTimeRecovery` | `false` | Enable continuous backups on the DynamoDB table. |
| `table.deletionProtection` | `false` | Block accidental table deletion at the API level. |

## Connect your MCP client

Add the `McpUrl` to your `.mcp.json`:

```json
{
  "mcpServers": {
    "kalem": {
      "type": "http",
      "url": "<McpUrl>",
      "headers": {
        "Authorization": "Bearer <your-api-key>"
      }
    }
  }
}
```

<Note>
  Keep your API key out of version control. If `.mcp.json` is committed, reference the token via an environment variable or a secrets manager rather than pasting the literal value. Revoke any leaked key immediately using the `revoke_api_key` MCP tool.
</Note>

## Connect the developer CLI

Log in with your `ApiUrl` and API key:

```bash
turjuman login --url <ApiUrl> --key <your-api-key>
```

This saves your credentials to `~/.turjuman/auth.json`. After logging in, you can run `turjuman init`, `turjuman pull`, and `turjuman push` as normal.

## Health check

Both Lambda Function URLs respond to an unauthenticated `GET /health`:

```bash
curl https://<McpUrl>/health
```

```json
{"status":"ok"}
```

A `200` response confirms the function is deployed and reachable. Use this in monitoring or uptime checks.

## Check deploy status

To see whether Kalem is installed in your AWS account and region:

```bash
npx @turjuman/aws-deploy status
```

This lists all Kalem-managed CloudFormation stacks in the configured region, their current status, and their stack outputs.

## Update

To deploy a new version of Kalem, run:

```bash
npx @turjuman/aws-deploy deploy
```

Because `turjuman.deploy.json` (or the SSM canonical copy) retains your settings, the redeploy is non-interactive and updates the Lambda code in-place.

## Teardown

<Warning>
  Teardown permanently deletes your CloudFormation stack and, by default, the DynamoDB table and all its data. This is irreversible.
</Warning>

To remove the stack interactively (you will be prompted to confirm by typing the stack name):

```bash
npx @turjuman/aws-deploy teardown
```

To skip the confirmation prompt, pass `--yes`. To target a specific stack by name, use `--stack-name`:

```bash
npx @turjuman/aws-deploy teardown --yes --stack-name my-kalem-stack
```

To preserve the DynamoDB table (and its data) while removing the Lambda functions and other resources:

```bash
npx @turjuman/aws-deploy teardown --keep-table
```

## Cost

Kalem uses on-demand DynamoDB billing and AWS Lambda, both of which have generous free tiers:

- **Lambda**: 1 million free requests and 400,000 GB-seconds of compute per month.
- **DynamoDB**: 25 GB of storage, 25 read capacity units, and 25 write capacity units free (or pay-per-request at fractions of a cent per million operations).

For most small teams managing translations, the monthly cost is effectively **$0**.

## Bootstrap an existing table

If you deployed the CloudFormation stack manually (without using `@turjuman/aws-deploy deploy`) or need to create the first owner against an already-deployed table, use the `bootstrap` command:

```bash
npx @turjuman/aws-deploy bootstrap \
  --table <TableName> \
  --region us-east-1 \
  --email you@example.com \
  --name "Your Name"
```

This creates the first OWNER and prints the API key once. Re-running the command when the org already has users is safe — it exits without making changes.
