Self-hosting Turjuman means running it in your own AWS account: one DynamoDB table and up to three Lambda Function URLs (MCP server, REST API, and webhook dispatcher). Turjuman ships as a standalone AWS CDK construct, **`@turjuman/aws-cdk`**, with the Lambda handlers pre-bundled and vendored into the package. You deploy it with the AWS CDK CLI (`cdk deploy`) — no SAM CLI and no clone of the Turjuman repo — then create the first owner over HTTP with `turjuman bootstrap`.

<Warning>
  The CDK stack is conventional but has **not yet been verified end-to-end against a live AWS account**. Use the [Quickstart](/quickstart) to evaluate Turjuman locally first, and treat the AWS deploy as beta. Please report any issues you encounter.
</Warning>

## Prerequisites

- An AWS account with credentials configured (via environment variables, `~/.aws/credentials`, or an IAM role)
- Node.js 24 or later
- The AWS CDK CLI: `npm install -g aws-cdk` (or use `npx cdk`)
- Sufficient IAM permissions to create CloudFormation stacks, Lambda functions, DynamoDB tables, and IAM roles

## Deploy

<Steps>
  <Step title="Create a CDK app">
    Make a small project that instantiates the `TurjumanStack`. The construct carries its own pre-bundled Lambda assets, so the only dependencies are the construct and the CDK runtime:

    ```bash
    mkdir turjuman-deploy && cd turjuman-deploy
    npm init -y && npm pkg set type=module
    npm install @turjuman/aws-cdk aws-cdk-lib constructs
    ```

    Add a `cdk.json` that tells the CDK CLI how to run your app:

    ```json
    {
      "app": "node app.mjs"
    }
    ```

    Then write `app.mjs`. Every prop is optional — the defaults reproduce the standard stack (on-demand DynamoDB, arm64 Lambdas at 256 MB, all three surfaces):

    ```js
    import { App } from "aws-cdk-lib";
    import { TurjumanStack } from "@turjuman/aws-cdk";

    const app = new App();
    new TurjumanStack(app, "Turjuman", {
      // stackName: "turjuman",
      // corsAllowOrigins: ["https://app.example.com"],
      // table: { pointInTimeRecovery: true, deletionProtection: true },
    });
    ```

    <Tip>
      Prefer to embed Turjuman in an existing stack? Import the `Turjuman` **construct** instead of `TurjumanStack` and drop it into any `Stack`. Its public members — `table`, `mcpFunction`, `mcpUrl`, `apiFunction`, `apiUrl`, `webhookFunction` — let you wire your own resources (grants, outputs, alarms) without any extra props.
    </Tip>
  </Step>

  <Step title="Bootstrap CDK and deploy">
    The default synthesizer publishes the Lambda assets through the standard CDK bootstrap bucket, so bootstrap the environment once per account/region, then deploy:

    ```bash
    cdk bootstrap        # once per account+region
    cdk deploy           # creates/updates the CloudFormation stack
    ```

    When the stack reaches `CREATE_COMPLETE`, the CDK CLI prints the [stack outputs](#stack-outputs):

    ```
    Outputs:
    Turjuman.McpUrl   = https://xxxx.lambda-url.us-east-1.on.aws/
    Turjuman.ApiUrl   = https://yyyy.lambda-url.us-east-1.on.aws/
    Turjuman.TableName = Turjuman-TurjumanTableXXXX-XXXXXXXXXXXX
    ```
  </Step>

  <Step title="Create the first owner">
    With the stack deployed, create the first OWNER over HTTP and capture its API key. This works only while the deployment has no users — a second call returns `409`:

    ```bash
    turjuman bootstrap --url <ApiUrl> --email you@example.com --name "Your Name"
    ```

    <Note>
      The API key is printed **once** and is not stored in plaintext (only its SHA-256 hash is). `turjuman bootstrap` also saves it to `~/.turjuman/auth.json`, so you're logged in for the CLI immediately — but copy it to a secrets manager now; it cannot be retrieved again.
    </Note>

    <Warning>
      `POST /v1/bootstrap` is **unauthenticated** — by design, since there is no key yet — and whoever calls it first becomes the global OWNER. The single-owner guard is atomic (concurrent calls can't both win, and a second call returns `409`), but it does **not** verify *who* the first caller is. **Bootstrap immediately after `cdk deploy`, before sharing the `ApiUrl`.** Until you do, anyone who learns the `ApiUrl` (it appears in deploy logs, CI output, screenshots) can claim ownership and lock you out — in which case tear the stack down (the table is retained) and redeploy.
    </Warning>

    `turjuman bootstrap` talks to the REST API surface, so it requires `api.enabled` (the default). If you deploy MCP-only (`api: { enabled: false }`), there is no bootstrap route — re-enable the REST API to create the owner, then disable it again if you wish.
  </Step>
</Steps>

## Stack outputs

| Output | Description |
|--------|-------------|
| `McpUrl` | The Lambda Function URL for the MCP server. Use this in your `.mcp.json`. |
| `ApiUrl` | The Lambda Function URL for the REST API. Use this with `turjuman bootstrap` and `turjuman login`. Absent when the REST API surface is disabled. |
| `TableName` | The DynamoDB table name, for manual operations. |

## Configuration

Configure the stack by passing props in your `app.mjs` — composition in code, not CLI flags. All props are optional.

```js
new TurjumanStack(app, "Turjuman", {
  stackName: "turjuman",
  corsAllowOrigins: ["https://app.example.com"],
  functionDefaults: { memorySize: 512, timeout: 30 },
  mcp: { memorySize: 1024 },            // per-function override
  api: { enabled: true },
  webhook: { enabled: true },
  table: { pointInTimeRecovery: true, deletionProtection: true },
});
```

| Prop | Default | Description |
|------|---------|-------------|
| `stackName` | construct id | CloudFormation stack name. |
| `corsAllowOrigins` | `["*"]` | Allowed origins for Function URL CORS. |
| `functionDefaults.memorySize` | `256` | Lambda memory (MB) applied to every function. |
| `functionDefaults.timeout` | `15` | Lambda timeout (seconds) applied to every function. |
| `functionDefaults.architecture` | `"arm64"` | CPU architecture (`"arm64"` or `"x86_64"`). |
| `mcp` / `api` / `webhook` | — | Per-function tuning that overrides `functionDefaults`. |
| `api.enabled` / `webhook.enabled` | `true` | Deploy that surface. Set `false` to remove it. |
| `webhook.streamStartingPosition` | `"LATEST"` | Where the Streams poller starts reading: `"LATEST"` (only changes made after deploy) or `"TRIM_HORIZON"` (from the shard start). |
| `table.billingMode` | `"PAY_PER_REQUEST"` | `"PROVISIONED"` uses `readCapacity`/`writeCapacity`. |
| `table.pointInTimeRecovery` | `false` | Continuous backups on the DynamoDB table. |
| `table.deletionProtection` | `false` | Block table deletion at the DynamoDB API level. |
| `vpc` | — | Run the functions inside an existing VPC (`subnetIds`, `securityGroupIds`). |

## Connect your MCP client

Add the `McpUrl` to your `.mcp.json`:

```json
{
  "mcpServers": {
    "turjuman": {
      "type": "http",
      "url": "<McpUrl>",
      "headers": {
        "Authorization": "Bearer <your-api-key>"
      }
    }
  }
}
```

<Note>
  Keep your API key out of version control. If `.mcp.json` is committed, reference the token via an environment variable or a secrets manager rather than pasting the literal value. Revoke any leaked key immediately using the `revoke_api_key` MCP tool.
</Note>

## Connect the developer CLI

`turjuman bootstrap` already logged you in. On another machine, log in with your `ApiUrl` and API key:

```bash
turjuman login --url <ApiUrl> --key <your-api-key>
```

This saves your credentials to `~/.turjuman/auth.json`. After logging in, you can run `turjuman init`, `turjuman pull`, and `turjuman push` as normal.

## Health check

The MCP Function URL answers an unauthenticated `GET /health`:

```bash
curl https://<McpUrl>/health
```

```json
{"status":"ok"}
```

A `200` confirms the function is deployed and reachable. The REST Function URL exposes an unauthenticated `GET /` returning the service name and version for the same purpose.

## Update

To deploy a new version of Turjuman, bump `@turjuman/aws-cdk` in your deploy project and redeploy — the Lambda code updates in place:

```bash
npm update @turjuman/aws-cdk
cdk deploy
```

## Teardown

```bash
cdk destroy
```

<Warning>
  The DynamoDB table carries `RemovalPolicy.RETAIN`, so `cdk destroy` deletes the Lambda functions and other resources but **leaves the table (and all its data) in place** — by design, so an accidental teardown can't lose data. To remove the data too, delete the table yourself afterwards:

  ```bash
  aws dynamodb delete-table --table-name <TableName>
  ```

  If you enabled `table.deletionProtection`, disable it first (`aws dynamodb update-table --table-name <TableName> --no-deletion-protection-enabled`).
</Warning>

## Cost

Turjuman uses on-demand DynamoDB billing and AWS Lambda, both of which have generous free tiers:

- **Lambda**: 1 million free requests and 400,000 GB-seconds of compute per month.
- **DynamoDB**: 25 GB of storage, 25 read capacity units, and 25 write capacity units free (or pay-per-request at fractions of a cent per million operations).

For most small teams managing translations, the monthly cost is effectively **$0**.
