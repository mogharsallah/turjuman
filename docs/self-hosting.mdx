Turjuman deploys to your own AWS account. The whole stack is **one DynamoDB table and up to three
Lambda functions** exposed as Function URLs (the MCP server is always deployed; the REST API and the
webhook dispatcher are optional). The infrastructure is the
[`@turjuman/aws-cdk`](https://github.com/mogharsallah/turjuman/tree/main/packages/aws-cdk) construct,
built on the [AWS CDK](https://docs.aws.amazon.com/cdk/). The `turjuman-aws-deploy deploy` command
self-bootstraps the CDK environment, deploys the stack from pre-bundled Lambda assets, and creates
your first owner — in one step.

<Note>
  Self-hosting lives in a **separate package, `@turjuman/aws-deploy`** (binary `turjuman-aws-deploy`),
  so the day-to-day [`turjuman` developer CLI](/reference/cli-commands) stays a lean install with
  no AWS SDK. Run the deploy tool with `npx @turjuman/aws-deploy <command>` — the Lambda code ships
  pre-bundled inside the published `@turjuman/mcp-server` / `@turjuman/api` packages, so you no longer
  need a clone of the repo to deploy.
</Note>

<Note>
  **No SAM CLI.** The CLI self-bootstraps the standard CDK environment (the shared `CDKToolkit`
  stack — a staging bucket + roles) on first run, so a deploy stays one command. Pass
  `--skip-bootstrap` if you bootstrap your accounts yourself (`cdk bootstrap`).
</Note>

<Warning>
  The CDK stack is conventional but has **not yet been verified end-to-end against a live AWS
  account** (see the [roadmap](https://github.com/mogharsallah/turjuman/blob/main/ROADMAP.md)).
  Use the [Quickstart](/quickstart) to evaluate, and treat the cloud deploy as beta.
</Warning>

## Prerequisites

- An AWS account and credentials configured (`aws configure`, or any standard AWS env vars).
- Node.js 20+.
- Permission to create the CDK bootstrap resources on first deploy (CloudFormation, S3, IAM, SSM) —
  or a pre-bootstrapped account plus `--skip-bootstrap`.

## Deploy

```bash
npx @turjuman/aws-deploy deploy   # interactive: stack name, region, options, first owner
```

`deploy` will:

1. self-bootstrap the standard CDK environment (idempotent — creates/updates the shared `CDKToolkit`
   stack), unless you pass `--skip-bootstrap`;
2. synthesize the CDK stack — with the pre-bundled Lambda assets — and deploy/update it with the CDK
   toolkit using your credentials, publishing the assets through the bootstrap staging bucket;
3. create the first owner and print the API key **once**, saving it to `~/.turjuman/auth.json` so
   you're immediately logged in.

The resolved configuration is stored as the **canonical copy in AWS SSM**
(`/turjuman/<stack>/deploy-config`) and cached locally in `turjuman.deploy.json`. Because the
canonical copy lives in AWS, re-running `npx @turjuman/aws-deploy deploy` updates the stack
non-interactively **from any machine or CI** — it reads SSM first, falling back to the local cache,
then prompts. Use `npx @turjuman/aws-deploy deploy --reconfigure` to re-prompt. (Owner email/name are
PII and are kept in the local cache only, never written to SSM.)

Outputs printed at the end:

| Output | Use |
|---|---|
| `McpUrl` | MCP server URL for your `.mcp.json` |
| `ApiUrl` | REST API URL for `turjuman login --url` (omitted if the API surface is disabled) |
| `TableName` | the DynamoDB table |

### Options

`deploy` prompts for these; all default to the values below (the prior topology):

| Setting | Default | Notes |
|---|---|---|
| CORS origins | `*` | Comma-separated list of allowed origins for the Function URLs |
| Lambda memory | `256` MB | |
| Lambda timeout | `15` s | |
| REST API surface | on | The CLI/CI REST API. Disable to run MCP-only |
| Webhook surface | on | The DynamoDB Streams → webhook dispatcher |
| Point-in-time recovery | off | Continuous DynamoDB backups |
| Deletion protection | off | Block table deletion at the API level |
| VPC | off | Optionally supply subnet + security-group ids to run the functions in a VPC |

Re-running `deploy` is the one path for any change. For non-interactive single-knob edits (handy in
CI), skip the prompts with flags — each reads the canonical config, applies the change, and
redeploys:

```bash
npx @turjuman/aws-deploy deploy --disable webhook        # remove a surface (api|webhook)
npx @turjuman/aws-deploy deploy --enable webhook         # add it back
npx @turjuman/aws-deploy deploy --set mcp.memorySize=512 # tune one function
npx @turjuman/aws-deploy deploy --set table.billingMode=PROVISIONED
```

<Note>
  The table's **Stream stays on even when the webhook surface is disabled**, so re-enabling webhooks
  later never replaces the table. Switching `table.billingMode` is subject to DynamoDB's
  once-per-24h limit, and provisioned `readCapacity`/`writeCapacity` (default 25/25) are split across
  the table and its three GSIs.
</Note>

### The first owner

The first owner is created directly against the new table (the org-already-populated guard lives in
core, so re-running `deploy` won't create a second owner — it keeps your existing key). The printed
`apiKey` is your MCP/CLI bearer token; from there you manage users, projects and keys through the MCP
server.

## Runtime configuration

The Lambda functions read these environment variables:

| Variable | Default | Effect |
|---|---|---|
| `TURJUMAN_TABLE` | (set by deploy) | DynamoDB table name. |
| `TURJUMAN_ALLOWED_ORIGIN` | `*` | The `Access-Control-Allow-Origin` the MCP server returns. Lock it to a single origin to restrict browser callers. |

On the managed AWS deploy the **CORS origins** prompt is the primary lock-down — it configures CORS
natively on the Function URL (which supersedes the function's own header), so set the allowed origins
there. `TURJUMAN_ALLOWED_ORIGIN` is the app-level equivalent for running the function behind your
own gateway or locally.

### Health check

The MCP server answers an unauthenticated `GET /health` with `200 {"status":"ok"}` — point a load
balancer or uptime monitor at it to confirm liveness without a JSON-RPC round-trip or an API key.
Every other non-`POST` verb returns `405` with an `Allow` header. Clients that send an unsupported
`MCP-Protocol-Version` header get a `400`; an absent header is treated as the `2025-03-26` baseline.

## Checking what's installed

To see whether Turjuman is already installed in an AWS account/region:

```bash
npx @turjuman/aws-deploy status                 # uses turjuman.deploy.json / AWS_REGION
npx @turjuman/aws-deploy status --region eu-west-1
```

`status` reports the stack named in `turjuman.deploy.json` (if present) **and** scans the region
for any Turjuman-managed stacks by tag — so it finds installs even from a fresh machine with no
local config, or when the stack was renamed. For each stack it reads the canonical config from SSM
and prints which surfaces/knobs are configured. Every `deploy` tags its stack `app=turjuman` /
`turjuman:managed=true`, which is what the scan keys off. CloudFormation is regional, so run it
once per region you want to check.

## Removing Turjuman (teardown)

To remove an installation entirely:

```bash
npx @turjuman/aws-deploy teardown               # uses turjuman.deploy.json
npx @turjuman/aws-deploy teardown --stack-name turjuman --region us-east-1
```

`teardown` deletes the CloudFormation stack (Lambdas, IAM roles, Function URLs, **and the DynamoDB
table with all translation data**) and removes the stack's SSM config parameter. It asks you to type
the stack name to confirm, then offers to remove the local `turjuman.deploy.json` and
`~/.turjuman/auth.json`. It **does not** touch the shared CDK bootstrap (`CDKToolkit`) stack or its
staging bucket, which other stacks may share. It's idempotent — re-running after a partial teardown
won't error.

<Warning>
  This **permanently destroys all data** unless you pass `--keep-table`. Export anything you need
  first. Pass `--yes` (with an explicit `--stack-name`) to skip the typed confirmation in automation.
</Warning>

### Keeping your data

To remove the install but **preserve the DynamoDB table and its data**:

```bash
npx @turjuman/aws-deploy teardown --keep-table
```

The table is declared with `RemovalPolicy.RETAIN`, so deleting the stack always leaves it in place;
`--keep-table` additionally enables deletion protection on the now-orphaned table. Your data is safe
and no longer managed by CloudFormation. (A full teardown, without `--keep-table`, deletes the
retained table explicitly.) To remove a kept table later, disable protection and delete it:

```bash
aws dynamodb update-table --table-name <TableName> --no-deletion-protection-enabled --region <region>
aws dynamodb delete-table --table-name <TableName> --region <region>
```

## Bootstrapping an owner against an existing table

If you provisioned the table some other way (or want a second org's owner), create an owner against
the deployed table directly:

```bash
npx @turjuman/aws-deploy bootstrap --table <TableName> --region <region> --email you@example.com --name "You"
```

This is the same client-side bootstrap `npx @turjuman/aws-deploy deploy` runs; it prints the API key
once and refuses to create a second owner once the org has users.

## Cost

- **DynamoDB**: on-demand (`PAY_PER_REQUEST`) by default — you pay per request, nothing when idle.
  Typical team usage stays within the free tier (25 GB storage + generous request allowances).
- **Lambda**: arm64, 256 MB, only billed per invocation; the free tier covers 1M requests/month.
- **Function URLs**: no API Gateway, no hourly cost.
- **CDK bootstrap**: a one-time staging S3 bucket (negligible — it only holds deploy assets).

For a small team this is effectively free to run.

## Local development

You can run everything locally against [LocalStack](https://www.localstack.cloud/) without an AWS
account — see the [Quickstart](/quickstart) and the
[contributing guide](/contributing#local-development). The helper script creates the table (matching
the deploy stack: PK/SK + GSI1/GSI2/GSI3, all `ALL` projection) and bootstraps an owner in one step:

```bash
npm run dev:setup you@example.com "You"
```

It's idempotent about the table (reuses an existing one). Alternatively, create the table yourself
and call `bootstrapOwner` from `@turjuman/core`.

## Updating

```bash
npx @turjuman/aws-deploy deploy   # re-runs non-interactively using the canonical SSM config
```

Pinning the deploy CLI version (`npx @turjuman/aws-deploy@<version> deploy`) ships a coherent
construct + MCP + API triple, so the code updates in place (URLs stay stable, the table is
untouched). DynamoDB schema changes are additive (new item types reuse the existing keys/GSIs), so
updates are normally just a redeploy.
