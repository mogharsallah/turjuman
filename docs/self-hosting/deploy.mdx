Turjuman ships as a standalone AWS CDK construct, **`@turjuman/aws-cdk`**, with the Lambda handlers pre-bundled and vendored into the package. You deploy it with the AWS CDK CLI (`cdk deploy`) ŌĆö no SAM CLI and no clone of the Turjuman repo ŌĆö then create the first owner over HTTP with `turjuman bootstrap`. Read the [Overview](/self-hosting/overview) first for the topology, prerequisites, and the beta caveat.

## Deploy

<Steps>
  <Step title="Create a CDK app">
    Make a small project that instantiates the `TurjumanStack`. The construct carries its own pre-bundled Lambda assets, so the only dependencies are the construct and the CDK runtime:

    ```bash
    mkdir turjuman-deploy && cd turjuman-deploy
    npm init -y && npm pkg set type=module
    npm install @turjuman/aws-cdk aws-cdk-lib constructs
    ```

    Add a `cdk.json` that tells the CDK CLI how to run your app:

    ```json cdk.json
    {
      "app": "node app.mjs"
    }
    ```

    Then write `app.mjs`. Every prop is optional ŌĆö the defaults reproduce the standard stack (on-demand DynamoDB, arm64 Lambdas at 256 MB, all three surfaces):

    ```js app.mjs
    import { App } from "aws-cdk-lib";
    import { TurjumanStack } from "@turjuman/aws-cdk";

    const app = new App();
    new TurjumanStack(app, "Turjuman", {
      // stackName: "turjuman",
      // corsAllowOrigins: ["https://app.example.com"],
      // table: { pointInTimeRecovery: true, deletionProtection: true },
    });
    ```

    See [Configuration](/self-hosting/configuration) for every prop. Prefer to embed Turjuman in an existing stack? Import the `Turjuman` **construct** instead of `TurjumanStack` and drop it into any `Stack`.
  </Step>

  <Step title="Bootstrap CDK and deploy">
    The default synthesizer publishes the Lambda assets through the standard CDK bootstrap bucket, so bootstrap the environment once per account/region, then deploy:

    ```bash
    cdk bootstrap        # once per account+region (standard CDK bootstrap)
    cdk deploy           # creates/updates the CloudFormation stack
    ```

    When the stack reaches `CREATE_COMPLETE`, the CDK CLI prints the [stack outputs](#stack-outputs):

    ```
    Outputs:
    Turjuman.McpUrl   = https://xxxx.lambda-url.us-east-1.on.aws/
    Turjuman.ApiUrl   = https://yyyy.lambda-url.us-east-1.on.aws/
    Turjuman.TableName = Turjuman-TurjumanTableXXXX-XXXXXXXXXXXX
    ```
  </Step>

  <Step title="Create the first owner">
    With the stack deployed, create the first OWNER over HTTP and capture its API key. This works only while the deployment has no users ŌĆö a second call returns `409`:

    ```bash
    turjuman bootstrap --url <ApiUrl> --email you@example.com --name "Your Name"
    ```

    <Note>
      The API key is printed **once** and is not stored in plaintext (only its SHA-256 hash is). `turjuman bootstrap` also saves it to `~/.turjuman/auth.json`, so you're logged in for the CLI immediately ŌĆö but copy it to a secrets manager now; it cannot be retrieved again.
    </Note>

    <Warning>
      `POST /v1/bootstrap` is **unauthenticated** by design (there is no key yet), so whoever calls it first becomes the global OWNER. Bootstrap immediately after `cdk deploy`, before sharing the `ApiUrl` ŌĆö see [Security & API keys](/self-hosting/security) for why this matters and what to do if someone beats you to it.
    </Warning>
  </Step>
</Steps>

## Stack outputs

| Output | Description |
|--------|-------------|
| `McpUrl` | The Lambda Function URL for the MCP server. Use this in your `.mcp.json`. |
| `ApiUrl` | The Lambda Function URL for the REST API. Use this with `turjuman bootstrap` and `turjuman login`. Absent when the REST API surface is disabled. |
| `TableName` | The DynamoDB table name, for manual operations. |

## Connect your MCP client

Add the `McpUrl` to your `.mcp.json`:

```json
{
  "mcpServers": {
    "turjuman": {
      "type": "http",
      "url": "<McpUrl>",
      "headers": {
        "Authorization": "Bearer <your-api-key>"
      }
    }
  }
}
```

For a client-by-client walkthrough (including Claude Code and code mode), see [Connect Claude Code](/guides/connect-claude-code).

<Note>
  Keep your API key out of version control. If `.mcp.json` is committed, reference the token via an environment variable or a secrets manager rather than pasting the literal value. Revoke any leaked key immediately with the `revoke_api_key` MCP tool.
</Note>

## Connect the developer CLI

`turjuman bootstrap` already logged you in on this machine. On another machine, log in with your `ApiUrl` and API key:

```bash
turjuman login --url <ApiUrl> --key <your-api-key>
```

This saves your credentials to `~/.turjuman/auth.json`. After logging in, you can run `turjuman init`, `turjuman pull`, and `turjuman push` as normal ŌĆö see [Sync with the CLI](/guides/sync-with-cli).

## Health check

The MCP Function URL answers an unauthenticated `GET /health`:

```bash
curl https://<McpUrl>/health
```

```json
{"status":"ok"}
```

A `200` confirms the function is deployed and reachable. The REST Function URL exposes an unauthenticated `GET /` returning the service name and version for the same purpose.

## Update

To deploy a new version of Turjuman, bump `@turjuman/aws-cdk` in your deploy project and redeploy ŌĆö the Lambda code updates in place:

```bash
npm update @turjuman/aws-cdk
cdk deploy
```

## Teardown

```bash
cdk destroy
```

<Warning>
  The DynamoDB table carries `RemovalPolicy.RETAIN`, so `cdk destroy` deletes the Lambda functions and other resources but **leaves the table (and all its data) in place** ŌĆö by design, so an accidental teardown can't lose data. To remove the data too, delete the table yourself afterwards:

  ```bash
  aws dynamodb delete-table --table-name <TableName>
  ```

  If you enabled `table.deletionProtection`, disable it first (`aws dynamodb update-table --table-name <TableName> --no-deletion-protection-enabled`).
</Warning>
