This page defines the terms used throughout Turjuman's documentation and tools. Field names (`baseLocale`, `approvedValue`, `stale`) are written exactly as they appear in the API. Each definition links to the page that explains the concept in full.

<Note>
  "Glossary" is also a Turjuman **feature** — a per-project list of preferred translations and do-not-translate terms that guides AI translation. This page is the documentation glossary; for the feature, see the **Glossary** entry below.
</Note>

| Term | Definition |
|---|---|
| **ADMIN** | A global role with broad administrative access to projects and users, but which cannot change OWNER-level roles. Acts as MANAGER on every project. See [Roles & permissions](/concepts/roles-and-permissions). |
| **API key** | A bearer token sent as `Authorization: Bearer <key>` on every request. Tied to a user, it inherits that user's roles and can be further scoped read-only or given an expiry. Stored as a SHA-256 hash. See [Security & API keys](/self-hosting/security). |
| **`approvedValue`** | The stable, delivery-ready snapshot of a translation, updated only when a translation is approved. `turjuman pull` exports this by default. See [Lifecycle](/concepts/lifecycle). |
| **Base locale** (`baseLocale`) | The source language a project's strings are authored in (e.g. `en`). Set when the project is created. See [Architecture](/concepts/architecture). |
| **Classic mode** | The default MCP connection mode, where every operation is advertised as its own tool. Contrast with code mode. See [How agents use Turjuman](/concepts/how-agents-use-turjuman). |
| **Code mode** | An MCP connection mode (`?mode=code`) that advertises only `search`, `describe`, and `run_code`, letting an agent run a sandboxed TypeScript program instead of many tool calls. See [Code mode](/guides/code-mode). |
| **Continuous localization** | The practice of moving translations with your code — pushing source strings and pulling approved translations on every relevant change, rather than in a separate release cycle. See [Lifecycle](/concepts/lifecycle). |
| **Delivery bundle** | The set of approved translations for a locale, exported as locale files by `turjuman pull` or the REST bundle endpoint. See [REST API](/reference/rest-api). |
| **Deprecated** | A key state meaning the key was absent from the most recent push for its namespace. Deprecated keys and their translations are retained, and restored to `active` if the key reappears. See [Lifecycle](/concepts/lifecycle). |
| **DEVELOPER** | A per-project role that can push and pull files, manage keys and locales, and write translations, but cannot approve them. Designed for CI pipelines. See [Roles & permissions](/concepts/roles-and-permissions). |
| **EDITOR** | A per-project role that can write and approve translations, manage keys and glossary terms, and add context. Designed for translators and AI agents. See [Roles & permissions](/concepts/roles-and-permissions). |
| **Glossary** | A per-project list of preferred translations and do-not-translate terms used to guide AI translation. Managed through the glossary MCP tools. See [MCP tools](/reference/mcp-tools). |
| **ICU MessageFormat** | The canonical format Turjuman stores translation values in, including plurals. Format adapters convert to and from each file format's native plural form. See [File formats](/reference/file-formats). |
| **Ingest** | The lifecycle stage where source strings are pushed into Turjuman (`turjuman push` or the REST import endpoint). New keys are created `untranslated`; changed base values flag translations `stale`. See [Lifecycle](/concepts/lifecycle). |
| **Key** (TranslationKey) | A single translatable string within a project, identified by its `namespace` and `name`. Carries metadata like `description`, `tags`, `maxLength`, and `plural`. See [Architecture](/concepts/architecture). |
| **Locale** | A target language added to a project (e.g. `fr`, `ja`), identified by its `code`. See [Architecture](/concepts/architecture). |
| **MANAGER** | A per-project role with full control of the project, including members, locales, and keys, plus writing and approving translations. See [Roles & permissions](/concepts/roles-and-permissions). |
| **MCP** (Model Context Protocol) | The protocol Turjuman's primary surface speaks. The MCP server exposes Turjuman's full management capability as tools an AI agent calls. See [Why MCP-first](/concepts/why-mcp-first). |
| **MEMBER** | The standard global role: can work within projects the user is explicitly added to, with no org-wide administrative privileges. See [Roles & permissions](/concepts/roles-and-permissions). |
| **Membership** | A record linking a user to a project with a specific per-project role. A user can hold different roles on different projects. See [Architecture](/concepts/architecture). |
| **Namespace** | A grouping of related keys within a project. Ingest and deprecation are namespace-scoped, so you can push one namespace at a time. See [Lifecycle](/concepts/lifecycle). |
| **Operation** | A single capability declared once in the shared service layer and projected to both the MCP tool surface and the REST API, so behaviour and permissions never diverge. See [Architecture](/concepts/architecture). |
| **Origin** | Read-only provenance on a translation recording how it was produced: `human`, `llm`, `tm`, `mt`, or `import`. See [Lifecycle](/concepts/lifecycle). |
| **OWNER** | The highest global role: full control of all projects, users, and settings, and the only role that can grant privileged global roles. At least one OWNER must always exist. See [Roles & permissions](/concepts/roles-and-permissions). |
| **Plural** | A key marked as having plural forms, stored canonically in ICU MessageFormat and converted to each format's native plural representation on export. See [File formats](/reference/file-formats). |
| **Project** | The top-level container for a set of translations, with an `id`, `name`, `slug`, and `baseLocale`. See [Architecture](/concepts/architecture). |
| **QA check** | A deterministic, advisory check (e.g. missing placeholders, length violations) run over translations. See [QA checks](/reference/qa-checks). |
| **Read-only key** | An API key created with `readOnly: true`, which can perform read operations only, regardless of the user's roles. Set at creation and immutable. See [Security & API keys](/self-hosting/security). |
| **REST API** | The narrow, file-shaped surface that backs the CLI and CI — list, export, and import operations only. Full management lives on the MCP server. See [REST API](/reference/rest-api). |
| **Stale** | A flag set automatically on every translation for a key when its base-locale value changes, signaling the translation may no longer match the source. Cleared on approval. See [Lifecycle](/concepts/lifecycle). |
| **Status** | A translation's workflow state, advancing `untranslated` → `translated` → `approved`. Editing an approved translation returns it to `translated`. See [Lifecycle](/concepts/lifecycle). |
| **Translation** | The string data for a key in a specific locale, holding the working `value` and the approved `approvedValue` plus `status`, `stale`, and `origin`. See [Architecture](/concepts/architecture). |
| **Translation memory** (TM) | A store of prior phrasings, searched (exact, normalized, and fuzzy) so previously approved translations can be reused. See [Translate with MCP](/guides/translate-with-mcp). |
| **`value`** | The working draft slot of a translation, updated on every write — by a human, an agent, or an import — independent of the approved snapshot. See [Lifecycle](/concepts/lifecycle). |
| **VIEWER** | A read-only per-project role: can read all project content but modify nothing. See [Roles & permissions](/concepts/roles-and-permissions). |
| **Webhook** | An outbound, HMAC-signed HTTP notification sent when translation events occur, dispatched from DynamoDB Streams. See [Webhooks & events](/guides/webhooks). |
