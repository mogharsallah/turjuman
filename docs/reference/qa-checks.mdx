Turjuman's QA engine validates translations deterministically and reports findings without making any changes to your data. It never auto-accepts a translation or modifies its lifecycle — every finding is advisory, and your team decides how to act on the results.

---

## Check catalogue

| ID | Default severity | What it catches |
|---|---|---|
| `icu_syntax` | error | Invalid ICU MessageFormat — unbalanced braces, malformed `plural`/`select` syntax, or unknown argument types |
| `placeholders` | error | The target's variable set differs from the base — a `{variable}` is missing from the translation or an extra one was introduced |
| `plural_forms` | error | A plural key is missing the required `other` form, or a key expected to be a plural expression is not ICU plural syntax |
| `markup` | warning | HTML or XML tags in the translation differ from those in the base value (missing tags, extra tags, or changed tag names) |
| `length` | warning | The translated value exceeds the key's configured `maxLength` character limit |
| `whitespace` | warning | Leading or trailing whitespace mismatch compared to the base, or consecutive whitespace characters in the value |
| `punctuation` | info | The terminal punctuation of the translation differs from the base (e.g. base ends with `.`, translation does not) |
| `empty` | error | A translation expected to carry a value (`proposed` or `accepted`) is blank |
| `glossary` | error / warning | A do-not-translate term was not preserved as-is in the translation (error); a preferred term from the glossary was not used (warning) |
| `duplicate` | info | Two or more keys share an identical non-empty value for the same locale |
| `stale` | info | The base (source) value has changed since this translation was last written |
| `coverage` | info | The key has no translation value for the locale — **disabled by default** |

## Running checks

You can run QA checks from three surfaces:

| Surface | How |
|---|---|
| MCP | Call `run_qa_checks` with `{ projectId, locale?, checks?, slot? }` |
| CLI | Run `turjuman check [--locale <code>] [--check <id...>]` |
| REST API | `POST /v1/projects/{id}/checks` |

The CLI exits with code `1` when any error-severity findings are present, making it easy to fail a CI step on quality regressions. You can also pass `--check` to `turjuman push` to run QA automatically after uploading.

---

## Report format

All three surfaces return findings in the same JSON structure:

```jsonc
{
  "projectId": "proj_…",
  "baseLocale": "en",
  "locales": ["fr", "de"],
  "checks": ["icu_syntax", "placeholders", "…"],
  "counts": { "error": 2, "warning": 5, "info": 1 },
  "findings": [
    {
      "checkId": "placeholders",
      "severity": "error",
      "namespace": "default",
      "keyName": "greeting",
      "localeCode": "fr",
      "message": "Placeholder mismatch: missing {name}"
    }
  ],
  "byLocale": {
    "fr": [ /* findings for fr */ ],
    "de": [ /* findings for de */ ]
  }
}
```

`counts` summarises how many findings exist at each severity level across all locales and checks. `byLocale` groups the same findings list by locale code for convenient per-language reporting.

<Note>
Findings are stable-sorted by locale code, then namespace, then key name, then check ID. This makes the output diffable in CI — the order is deterministic across runs.
</Note>

## Per-project configuration

You can customise which checks run and how findings are reported on a per-project basis using `set_qa_config` (MCP) or the REST API.

### Per-check overrides

For each check ID, you can override `enabled` and/or `severity`:

```jsonc
{
  "checks": {
    "coverage": { "enabled": true },
    "punctuation": { "enabled": false },
    "duplicate": { "severity": "warning" }
  }
}
```

Setting `enabled: false` removes the check entirely from reports. Setting a custom `severity` remaps all findings from that check to the new level (valid values: `"error"`, `"warning"`, `"info"`).

### Ignore rules

You can suppress specific findings without disabling the whole check. Each ignore rule must include at least one of the following fields:

| Field | Effect |
|---|---|
| `checkId` | Ignore findings from this check ID |
| `namespace` | Ignore findings in this namespace |
| `keyName` | Ignore findings for this key name |
| `locale` | Ignore findings for this locale code |

Fields within a single rule are combined with AND logic. Multiple rules are combined with OR logic.

```jsonc
{
  "ignores": [
    { "checkId": "punctuation", "locale": "ja" },
    { "keyName": "legacy.onboarding.step1" },
    { "namespace": "admin", "checkId": "length" }
  ]
}
```

<Tip>
Use ignore rules to acknowledge known exceptions (e.g. a legacy key that intentionally differs) rather than disabling an entire check globally. This keeps your reports accurate and actionable.
</Tip>
