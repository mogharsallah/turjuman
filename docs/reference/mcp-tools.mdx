Turjuman's MCP is a single **code surface**. Instead of advertising one tool per capability, it exposes exactly three tools and lets the agent write code against the whole operation catalogue. Every call is authenticated with `Authorization: Bearer <api-key>`, and every operation runs through the same shared service layer ‚Äî and the same authorization ‚Äî as the REST API.

## The three tools

| Tool | Purpose |
|---|---|
| `search` | Search one knowledge base spanning the operations (with typed signatures) **and** the docs (guides, concepts, reference); results are segmented by kind with `total`/`hasMore`. Pass a `query` (narrow with `kind`), or omit it for a cold-start orientation. |
| `describe` | Full detail for one result `id` ‚Äî an operation's typed input fields, output, signature, and hints, or a whole docs page. |
| `run_code` | Run TypeScript that calls `await turjuman.<operation>(args)` and returns a value; the sandbox has no network or filesystem, and only the final result returns. |

The pattern is **search ‚Üí describe ‚Üí run_code**: find the operation, confirm its exact arguments, then write the program. The operation `id`s that `search` returns (`op:<name>`) match the operation names in the catalogue below, and every one is callable from `run_code`. See the [MCP guide](/guides/code-mode) for worked examples and the `run_code` result shape.

## The operation catalogue

These are the operations you call from `run_code` (and that the REST API projects where they are file-shaped). Discover them at runtime with `search`/`describe`; this catalogue is the human-readable index.

### Projects & locales

| Operation | Purpose |
|---|---|
| `list_projects` | List all projects you can access |
| `get_project` | Fetch a single project by ID |
| `create_project` | Create a new project (requires `baseLocale`) |
| `update_project` | Update project settings ‚Äî name, base locale, and `requireHumanAccept` |
| `list_locales` | List all locales configured for a project |
| `add_locale` | Add a new target locale to a project |

### Branches

| Operation | Purpose |
|---|---|
| `list_branches` | List a project's branches (`main` always exists) |
| `get_branch` | Fetch a single branch by ID |

### Namespaces

| Operation | Purpose |
|---|---|
| `list_namespaces` | List a project's namespaces |
| `get_namespace` | Fetch a single namespace by ID or name |
| `create_namespace` | Create a namespace (an optional grouping of keys) |
| `update_namespace` | Update a namespace's title, description, or lifecycle |

### Keys

| Operation | Purpose |
|---|---|
| `list_keys` | List keys in a project (paged: `limit` + `nextCursor`) |
| `search_keys` | Full-text search across key names and descriptions (paged) |
| `get_key` | Fetch a single key and its translations, by name (optionally within a namespace) |
| `create_key` | Create a key ‚Äî optionally set `namespace`, `description`, `tags`, `maxLength`, `plural`, `baseValue` |
| `update_key` | Update key metadata (description, tags, maxLength, plural) |
| `rename_key` | Rename or re-namespace a key, preserving its id and all translations |
| `delete_key` | Hard-delete a key and all its translations ‚Äî requires `confirm: true` |

### Translations

| Operation | Purpose |
|---|---|
| `get_translations` | Fetch translations by key (all locales) or by locale (paged) |
| `list_untranslated` | List keys that have no value for a given locale (paged) |
| `list_stale` | List cells whose source value has changed since they were written (paged) |
| `set_translation` | Write a working draft `value` for a key/locale; the cell becomes `proposed` |
| `bulk_set_translations` | Write up to 500 draft entries in one call; unknown keys are silently skipped |
| `accept_translation` | Accept a proposed value: append a version and advance the cell's `head` (a compare-and-swap). Honors `requireHumanAccept` |

### Runs

A **run** is the controlled write primitive that records what an agent did across a set of cells ‚Äî it gives idempotency, resumability, and a recorded budget.

| Operation | Purpose |
|---|---|
| `start_run` | Open a run on a branch (records `trigger`, `valueSource`, `idempotencyKey`) |
| `get_run` | Fetch a run's status and progress |
| `list_runs` | List runs for a project or branch |
| `finish_run` | Close a run, recording its final status and `budgetSpent` |
| `cancel_run` | Cancel an in-flight run |

### Glossary

| Operation | Purpose |
|---|---|
| `list_glossary` | List all glossary terms for a project |
| `add_glossary_term` | Add a new term to the glossary |
| `update_glossary_term` | Update an existing glossary term |
| `remove_glossary_term` | Remove a term from the glossary |

### Quality (QA)

| Operation | Purpose |
|---|---|
| `run_qa_checks` | Run advisory checks ‚Äî accepts `{ projectId, locale?, checks?, slot? }`, where `slot` is `working` or `accepted` |
| `get_qa_config` | Read the per-project QA configuration |
| `set_qa_config` | Write the per-project QA configuration (enable/disable checks, remap severities, add ignores) |

### Webhooks & lifecycle

| Operation | Purpose |
|---|---|
| `list_webhooks` | List all webhooks registered for a project |
| `add_webhook` | Register a new webhook ‚Äî the signing secret is returned only once at creation time |
| `remove_webhook` | Remove a registered webhook |
| `delete_project` | Permanently delete a project and all its data ‚Äî requires `confirm: true` |

### Administration

| Operation | Purpose |
|---|---|
| `list_users` | List users in your organisation |
| `create_user` | Create a new user account |
| `set_user_role` | Set a user's global role ‚Äî OWNER-gated for privileged roles |
| `list_members` | List members of a project |
| `add_member` | Add a user to a project |
| `set_member_role` | Change a project member's role |
| `remove_member` | Remove a user from a project |
| `create_api_key` | Create a new API key |
| `list_api_keys` | List all API keys in your organisation |
| `revoke_api_key` | Permanently revoke an API key |

## Paging

List operations support cursor-based pagination. Pass `limit` (default `100`, max `200`) to control page size. When there are more results, the response includes a `nextCursor` ‚Äî pass it back as `cursor` in your next call. Continue until `nextCursor` is absent or `null`.

```json
// First call
{ "projectId": "proj_xxx", "limit": 50 }

// Subsequent call
{ "projectId": "proj_xxx", "limit": 50, "cursor": "eyJpZCI6Mn0" }
```

## Authorization scales with the key

The mode never changes the *permissions*. Whether an operation is called directly or from inside a `run_code` sandbox, the request is dispatched to the same handler with the same authenticated context, and authorization is enforced in the core ‚Äî not merely hidden from a tool list.

- A **read-only key** (`readOnly: true`) can only perform read operations (`list_*`, `get_*`, `search_*`); a write is denied in the core.
- A key inherits its user's roles, so a key with global **MEMBER** role cannot invoke org-level administration operations.

See [Roles & permissions](/concepts/roles-and-permissions) for the underlying model.

## Operation hints

Each operation declares behaviour hints in its schema, visible through `describe`, so an agent framework can make safer decisions (for example, confirming before a destructive action).

| Property | Operations where set | Meaning |
|---|---|---|
| `readOnlyHint: true` | All `list_*`, `get_*`, `search_*` | The operation does not modify any server state |
| `destructiveHint: true` | `delete_key`, `delete_project`, `remove_webhook`, `revoke_api_key`, `remove_member`, `remove_glossary_term` | The operation permanently removes data |
| `idempotentHint: true` | e.g. `set_translation`, `set_qa_config`, `set_user_role`, `set_member_role` | Calling it again with the same arguments produces the same result |

<Note>
When an operation hits a validation or API error, it returns the error message as MCP `content` with `isError: true` rather than throwing. Your agent can read the error description and adjust its next action without the call itself failing the chain.
</Note>
