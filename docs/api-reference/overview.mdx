The Kalem REST API is the backend that powers the `kalem` CLI and your CI pipelines. It exposes a focused set of file-shaped operations â€” listing projects, exporting delivery bundles, and importing locale files â€” so that automated workflows always get predictable, deterministic responses. The base URL is the `ApiUrl` printed when you deploy your Kalem instance.

<Note>
  The REST surface is intentionally narrow. It covers list, export, and import operations only. For full project management â€” creating or updating projects, translating strings, managing members, or working with your glossary â€” use the [MCP server](/reference/mcp-tools) instead.
</Note>

## Authentication

Every route under `/v1/projects` requires a bearer token. Pass your API key in the `Authorization` header:

```bash
curl https://<ApiUrl>/v1/projects \
  -H "Authorization: Bearer op_live_..."
```

The following meta endpoints are public and do not require authentication:

- `GET /`
- `GET /v1/formats`
- `GET /v1/openapi.json`

| Response | Meaning |
|---|---|
| `401 Unauthorized` | The API key is missing, invalid, or expired |
| `403 Forbidden` | The key is valid but does not have permission for this resource |

## Base URL

Your base URL is the `ApiUrl` value printed by the deploy command:

```bash
npx @kalem/deploy deploy
# ...
# ApiUrl: https://kalem.example.com
```

If you've already authenticated with the CLI, you can also retrieve it with:

```bash
kalem login --url
```

All paths in this reference are relative to that base URL.

## Available endpoints

| Method | Path | Description |
|---|---|---|
| `GET` | `/` | Service metadata |
| `GET` | `/v1/formats` | List supported file formats |
| `GET` | `/v1/projects` | List projects you can access |
| `GET` | `/v1/projects/{id}` | Get a project |
| `GET` | `/v1/projects/{id}/locales` | List locales |
| `POST` | `/v1/projects/{id}/locales` | Add a locale |
| `GET` | `/v1/projects/{id}/keys` | List keys (supports `namespace`, `limit`, `cursor`) |
| `POST` | `/v1/projects/{id}/keys/import` | Import keys (the `kalem push` operation) |
| `GET` | `/v1/projects/{id}/translations` | List translations for a locale |
| `GET` | `/v1/projects/{id}/bundle` | Export delivery bundle for a locale |
| `POST` | `/v1/projects/{id}/translations/import` | Bulk-write translation values |
| `POST` | `/v1/projects/{id}/checks` | Run advisory QA checks |
| `GET` | `/v1/projects/{id}/qa-config` | Get QA config |
| `PUT` | `/v1/projects/{id}/qa-config` | Update QA config |

## Error format

All error responses return a JSON envelope with the same structure:

```json
{
  "error": "Human-readable message",
  "code": "ERROR_CODE",
  "requestId": "req_01j..."
}
```

Include the `requestId` when reporting issues â€” it lets you correlate the request with server-side logs.

<Tip>
  The full interactive API reference is available as an OpenAPI spec at `GET /v1/openapi.json` on your own deployment. You can load it into any OpenAPI-compatible tooling, such as Swagger UI or Postman.
</Tip>
